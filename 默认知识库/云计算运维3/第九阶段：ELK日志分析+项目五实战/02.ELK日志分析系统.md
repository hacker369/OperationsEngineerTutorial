## ES 集群部署

集群部署主要注意以下几个方面

集群配置参数：

* 节点发现机制
* ES 8.x 默认使用 Cluster Coordination Service（CCS）进行节点发现和集群形成，替代了之前的 Zen Discovery
* 配置项：

```shell
# 指定初始master节点（集群首次启动时必须配置）
cluster.initial_master_nodes:
  - node1
  - node2
  - node3

# 配置主机发现列表（推荐静态设置）
discovery.seed_hosts:
  - 192.168.126.211
  - 192.168.126.212
  - 192.168.126.213
  - 192.168.126.214
  - 192.168.126.215
  - 192.168.126.216

# 设置发现时等待超时时间（默认30秒，可调整）
discovery.find_peers_interval: 30s
```

避免脑裂问题：

* 为了防止脑裂（split-brain）现象，需要设置以下参数：

```shell
# 集群中需要选举master的最少节点数，推荐(N / 2) + 1
cluster.voting_only_nodes: 3

--------------------------------------------------------------------------------
这表示集群中设置了3个“仅投票节点”(voting-only nodes)。这些节点不作为master候选，只参与master选举的投票过程。
1. 什么是master选举的最小节点数？
ES的master选举依赖仲裁机制，推荐设置最小master数为(N / 2) + 1，其中N是投票节点（包含master-eligible节点和voting-only节点）的总数。
node.roles: [master]		# 主节点

2. voting_only_nodes是什么？
这些节点：
不会当选为master（即使master节点宕机，它们不会接管）
只用于维持集群仲裁，提高选举容错能力
node.roles: [voting_only]	# 投票节点
```

集群节点设置

ES 8.x 支持以下节点类型，建议按业务需求分配：

* master 节点：负责集群管理和协调，建议单独部署
* data 节点：存储和处理索引数据
* coordinating 节点（仅协调节点）：用于客户端负载均衡（选用场景）
* Ingest 节点：用于预处理数据

```shell
# master节点
node.master: true
node.data: false
node.ingest: false
node.roles: [ "master" ]

# data节点
node.master: false
node.data: true
node.ingest: false
node.roles: [ "data" ]
```

注意：

* master 节点与 data 节点分离：master 需保证稳定性，data 节点可能因高 IO 负载而不稳定
* 数量建议：建议至少 3 个 master 节点（避免脑裂），data 节点根据数据量规划

内存设置：

* ES 默认堆内存大小为 1GB，需要根据实际业务进行调整：

```shell
# 比如服务器内存为32GB，设置堆内存大小为系统内存的一半，但不能超过32GB
export ES_JAVA_OPTS="-Xms16g -Xmx16g"
```

修改配置文件

```shell
# vim /etc/elasticsearch/jvm.options
...
-Xms16g
-Xmx16g
```

注意：

* 堆内存分配过大会导致垃圾回收延迟过高，建议不超过 32GB
* JVM 预留的直接内存大小可以通过 `-XX:MaxDirectMemorySize`参数调整

硬盘空间规划

* 数据路径
* ES 的数据默认存储在 `/var/lib/elasticsearch`，可以通过配置文件修改：

```shell
path.data: /var/lib/elasticsearch

```

磁盘性能优化

* 挂载高性能存储（如 NVMe SSD）
* 启用数据压缩

```shell
index.codec: best_compression
```

配置磁盘分层存储（冷热分层）：在集群中配置冷热节点，热节点处理实时写入，冷接点处理长期存储：

## ES 基础概念

主要的基础概念有：Node、Index、Type、Document、Field、shard 和 replicas。

Node（节点）：运行单个 ES 实例的服务器

Cluster（集群）：一个或多个节点构成集群

Index（索引）：索引是多个文档的集合（办公室的柜子）

Type（类型）：一个 Index 可以定义一种或多种类型，将 Document 逻辑分组（柜子中文件夹，每个文件夹放置不同类型资料）

Document（文档）：Index 里面每条记录成为 Document，若干文档构建一个 Index（相当于文件夹中具体的资料，类似一张 A4 纸）

Field（字段）：ES 存储的最小单元（一张 A4 纸有很多列，每一列就相当于一个 Field）

Shards（分片）：ES 将 Index 分为若干份，每一份就是一个分片（办公室中有很多柜子）

Replicas（副本）：Index 的一份或多份副本（相当于对柜子中的数据进行备份）

为了便于理解，我们和 MySQL 这种关系型数据库做一个对比：

| 关系型数据库（MySQL、Oracle 等） | ES |
| --- | --- |
| database 或 schema | index |
| table | type |
| row | document |
| column 或 field | field |

ES 是分布式搜索引擎，每个索引有一个或多个分片（shard），索引的数据被分配到各个分片上。你可以看作是一份数据分成了多份给不同的节点。

当 ES 集群增加或删除节点时，shard 会在多个节点中均衡分配。默认是 5 个 primary shard（主分片）和 1 个 replica shard（副本，用于容错）。

# 三、ES 基本 API 操作（重点、记住）

前面我们通过 <http://192.168.126.212:9200/_cluster/health?pretty> 查看 ES 集群状态，其实就是它的一种 API 操作。

> 什么是 API？
>
> API（Application Programming Interface）应用程序编程接口，就是无需访问程序源码或理解内部工作机制就能实现一些相关功能的接口。

所谓的 API 就是日常生活中的 URL 地址 => 别人编写好程序或代码，我们不需要了解其底层原理，只需要通过 URL 获取数据即可。通常返回数据格式有两种：XML 格式与 JSON 格式。

## RestFul API 格式

```shell
# curl -X<verb> '<protocol>://<host>:<port>/<path>?<query_string>' -d '<body>'
```

| 参数 | 描述 |
| --- | --- |
| verb | http 方法，比如：get、post、put、delete、head 等 |
| host | ES 集群中的任意节点主机名 |
| port | ES HTTP 服务端口，默认 9200 |
| path | 索引路径 |
| query\_string | 可选的查询请求参数。例如 ?pretty 参数将返回 json 格式数据 |
| -d | data 缩写，里面放一个 json 格式请求主体 |
| body | 自己写的 json 格式的请求主体 |

ES 的 API 有很多，我们运维人员主要用到以下几个要介绍的较简单的 API。

更多 API 参考：https://www.elastic.co/guide/en/elasticsearch/reference/8.17/index.html

普及一个概念：

说明：ES 中主节点与数据节点关系和之前学习的 MySQL/Redis 中主从关系有所不同。

**主节点（master node）：**

* 负责管理和协调整个集群
* 决定如何分配数据
* 通常不负责处理数据的存储和搜索

**数据节点（data node）：**

* 实际存储数据和执行搜索操作
* 处理所有数据的增删改查请求

在小型或测试环境中，一个节点可能既是主节点也是数据节点。但在更复杂的环境中，主节点和数据节点通常是分开的，以确保效率和稳定性。主节点关注管理，而数据节点负责数据。

**match**

* 基本搜索查询，针对特定字段或字段集合进行搜索（类似**模糊匹配**，底层基于全文搜索）

```shell
特殊：match默认属于模糊匹配，但是如果是数字类型的匹配，相当于精准匹配，只有查询值为字符串类型，则相当于模糊匹配。
```

**bool**

```shell

            {"match":{"address":"lane"}}
          ]
        }
      }
    }
  '
```

**range**

* 指定区间内的数字或者时间
* 操作符：gt 大于，gte 大于等于，lt 小于，lte 小于等于

```shell
查询余额大于或等于20000且小于等于30000的账户

  -X GET "http://192.168.126.212:9200/bank/_search?pretty" \

                "lte":30000
              }
            }
          }
        }
      }
    }
  '
```

## logstash 简介

logstash 是一个开源的数据采集工具，通过数据源采集数据。然后进行过滤，并自定义格式输出到目的地。

数据分为：

1. 结构化数据，如：MySQL 数据库里的表等

### <font style="color:rgb(51, 51, 51);">采集 messages 日志</font>

logstash 配置文件，如 test.conf、messages.conf，专业名词来说叫做 pipeline（logstash）

<font style="color:rgb(51, 51, 51);">这里以 /var/log/messages 为例，让 logstash 进行日志采集，只定义 input 输入和 output 输出，不考虑过滤。最终将输出的日志输出到 ES 中！</font>

```shell
# 定义配置文件
logstash# cd /usr/share/logstash/bin
logstash# vim /etc/logstash/conf.d/messages.conf
input {
    file {
        path => "/var/log/messages"
        start_position => "beginning"
    }
}

output {
    elasticsearch{
        hosts => ["http://192.168.126.212:9200"]
        index => "messages-%{+YYYY.MM.dd}"
    }
}


# 启动logstash开始采集日志，注意先把之前启动的给停止掉！！！
logstash# nohup ./logstash --path.settings /etc/logstash/ -r -f /etc/logstash/conf.d/messages.conf &

后台运行如果要杀掉,请使用pkill java或ps查看PID再kill -9清除
```

### <font style="color:rgb(51, 51, 51);">采集多日志源</font>

我们使用 logstash 同时采集 messages.log 和 dnf.log 的日志信息，最终输出到 ES 服务器中。

```shell

input {
    file {
        path => "/var/log/messages"
        start_position => "beginning"
        type => "messages"
    }

    file {
        path => "/var/log/dnf.log"
        start_position => "beginning"
        type => "dnf"
    }
}

filter {

}

output {
    if [type] == "messages" {
        elasticsearch {
            hosts => ["http://192.168.126.212:9200"]
            index => "messages-%{+YYYY-MM-dd}"
        }
    }

    if [type] == "dnf" {
        elasticsearch {
            hosts => ["http://192.168.126.212:9200"]
            index => "dnf-%{+YYYY-MM-dd}"
        }
    }
}

# 检测上面的配置文件是否有语法问题
logstash# ./logstash --path.settings /etc/logstash -f /etc/logstash/conf.d/test.conf -t

# 启动logstash开始采集，注意先把之前启动的给停止掉！！！
logstash# nohup ./logstash --path.settings /etc/logstash/ -r -f /etc/logstash/conf.d/test.conf  &
```

## logstash 服务器操作

第一步：实际工作中，Nginx 肯定是会被各个地方的用户去访问的，所以后续我们会收集到 Nginx 的访问日志，来分析哪里的用户访问的最多等信息！所以，需要安装 IP => 区域地址 的插件。

```shell
logstash# gem sources --add https://mirrors.aliyun.com/rubygems/

```

放置 GeoLite2-City.mmdb 数据库到 /etc/logstash 目录（资料中已提供该数据库文件）

```shell
# ll /etc/logstash/GeoLite2-City.mmdb
```

> 这个插件：主要用于 IP 地址转 City，功能有二：① 可以把 IP 转经纬度 ② 可以把 IP 转换为具体城市，如背景、巴黎等。

第二步：编写 logstash pipeline 文件，采集 Nginx 日志信息到 ES 中

`vim /etc/logstash/conf.d/nginx.conf`

```shell
input {
    file {
        path => "/var/log/nginx/access.log"
        start_position => "beginning"
    }
}

filter {
    json {
        source => "message"
    }

    if [client_ip] =~ /,/ {
        mutate {
            split => { "client_ip" => "," }
            add_field => { "real_ip" => "%{client_ip[0]}" }
        }
    } else {
        mutate {
            add_field => { "real_ip" => "%{client_ip}" }
        }
    }

    geoip {
        source => "real_ip"
        target => "geoip"
        database => "/etc/logstash/GeoLite2-City.mmdb"
        fields => ["country_name", "region_name", "city_name", "location"]
    }

    date {
        match => [ "@timestamp", "ISO8601" ]
    }
}

output {
    elasticsearch {
        hosts => ["192.168.126.212:9200"]
        index => "nginx-%{+YYYY.MM.dd}"
    }
}
```

第三步：验证 pipeline 文件


第四步：将我们 Nginx 的访问日志替换为资料中提供的访问日志，目的是我们自己的 Nginx 访问日志，里面的 IP 都是局域网 IP，解析不出什么对应的地址，而资料中提供的 Nginx 访问日志，里面的 IP 基本都是外网 IP！可以通过 logstash 的插件去分析 IP 对应的地址信息。

# 八、扩展：filebeat

## Filebeat 介绍

作用：与 logstash 基本等价

ELK = ElasticSearch + Logstash + Kibana

EFK = ElasticSearch + Filebeat/Fluentd + Kibana

EFLK = ElasticSearch + Filebeat/Fluentd + Logstash + Kibana

因为 logstash 消耗内存等资源太高，如果在要采集日志的服务器上都安装 logstash，这样对应用服务器的压力会增加。所以我们要用轻量级的采集工具才更高效，更省资源。

![1766410647456-2b490331-9c48-458d-bc31-f1a45cd5d5a8.png](./img/Y5Iu6aq2YPD7gxdW/1766410647456-2b490331-9c48-458d-bc31-f1a45cd5d5a8-458647.png)

大概流程就是：

1. Filebeat/Fluentd 负责采集应用的日志信息，安装在业务服务器上
2. logstash 单独安装到服务器上，负责对第 1 步采集到的日志信息进行清洗、转化操作
3. ES 存储第 2 步得到的日志数据
4. Kibana 负责对 ES 中存储的日志数据进行分析、可视化展示

***

beats 是 ELK 官方推出的轻量级的日志收集处理工具，beats 占用资源少。

* Packetbeat：网络数据，收集网络流量数据
* Metricbeat：指标，收集系统、进程和文件系统级别的 CPU、内存使用情况等数据
* Filebeat：文件，收集日志文件数据
* Winlogbeat：Windows 事件日志，收集 Windows 事件日志数据
* Auditbeat：审计数据，收集审计日志
* Heartbeat：运行时间监控，收集系统运行时的数据

我们这里主要是收集日志信息，所以只讨论 filebeat。

filebeat 可以直接将采集的日志数据传输给 ES 集群（EFK），也可以给 logstash（5044 端口接收）。

![1766411470427-1ac28f73-516d-499c-9c7b-0f902565f9f2.png](./img/Y5Iu6aq2YPD7gxdW/1766411470427-1ac28f73-516d-499c-9c7b-0f902565f9f2-511653.png)

## EFK 搭建

EFK = ES + Filebeat + Kibana

Filebeat 负责收集应用服务器的日志信息，将日志信息直接传输存储到 ES 中，Kibana 负责分析、可视化 ES 中的日志数据。

第一步：克隆一台最小化安装的服务器，命名为 filebeat，内存 1G。

第二步：下载并安装 filebeat，下载地址：<font style="color:#117CEE;">https://www.elastic.co/cn/beats/filebeat</font>

```shell
# 使用资料中提供的安装包即可
filebeat# rpm -ivh filebeat-8.16.0-x86_64.rpm
```

第三步：配置 filebeat 收集日志数据

```yaml
filebeat# cat /etc/filebeat/filebeat.yml | grep -v '#' | grep -v '^$'
filebeat.inputs:
- type: filestream
  id: custom-logs
  enabled: true
  paths:
    - /var/log/*.log
    
  # 关闭所有可能自动设置数据流的模块
  filebeat.config.modules.enabled: false
  
  # -----------------------------彻底禁用模板和ILM---------------------------------
  # 禁用内置模板
  setup.template.enabled: false
  setup.template.overwrite: true
  
  # 禁用ILM
  setup.ilm.enabled: false
  setup.ilm.rollover_alias: ""
  setup.ilm.pattern: ""
  setup.ilm.policy_name: ""
  
  # -------------------------ES输出------------------------------
  output.elasticsearch:
    # 主机配置
    hosts: ["http://192.168.126.212:9200"]
    
    # 使用与filebeat完全不同的索引名前缀，避免触发数据流机制
    index: "filebeat1-%{+yyyy.MM.dd}"
    
    # 显式禁用ILM
    ilm.enabled: false
    
    # 禁用任何自动索引设置
    indices:
      - index: "filebeat1-%{+yyyy.MM.dd}"
        lifecycle:
          enabled: false
          name: ""
        data_stream:
          enabled: false
```

第四步：启动服务

```shell
filebeat# systemctl start filebeat
filebeat# systemctl enable filebeat
```

## ELFK 搭建

ELFK = ES + Logstash + Filebeat + Kibana

Filebeat 负责采集日志数据，然后交给 Logstash

Logstash 负责将日志数据进行过滤处理，将结果存储到 ES 中

ES 负责存储日志数据

Kibana 负责对日志进行过滤筛选展示

***

第一步：在 Logstash 上要重新配置，开放 5044 端口给 filebeat 连接，并重启 Logstash 服务

5044：logstash 默认端口

filebeat 可以把日志发送到 logstash 节点对应的 5044 端口，这样 logstash 就可以获取到 filebeat 采集的日志信息。

```shell
logstash# pkill java
logstash# vim /etc/logstash/conf.d/filebeat.conf
input {
  beats {
    port => 5044
  }
}

# 如果需要设置时区，在此处添加过滤器
filter {
  # 创建一个具有上海时区的时间戳字段供索引使用
  ruby {
    code => "event.set('indexDate', Time.now.getlocal('+08:00').strftime('%Y.%m.%d'))"
  }
}

output {
  elasticsearch {
    hosts => ["192.168.126.212:9200"]
    index => "filebeat2-%{indexDate}"
  }
  stdout {
    
  }
}

logstash# cd /usr/share/logstash/bin

如果前面有使用后台跑过logstash实例的请kill掉先
logstash# pkill java
logstash# ./logstash --path.settings /etc/logstash/ -r -f /etc/logstash/conf.d/filebeat.conf

目前是前台启动运行，可以看到测试效果，如果后期需要后台运行，可以使用 nohup + & 后台运行符
logstash# nohup ./logstash --path.settings /etc/logstash/ -r -f /etc/logstash/conf.d/filebeat.conf &
```

第二步：配置 filebeat 收集日志

```yaml
filebeat# cat /etc/filebeat/filebeat.yml | grep -v '#' | grep -v '^$'
filebeat.inputs:
- type: log
  enabled: true					# 改为true
  paths:
    - /var/log/*.log		# 收集的日志路径

filebeat.config.modules:
  path: ${path.config}/modules.d/*.yml
  reload.enabled: false
  
# 禁用模板设置，避免数据流问题
setup.template.enabled: false
# setup.template.settings:
# 	index.number_of_shards: 1

# 禁用Kibana自动配置
setup.kibana.enabled: false

output.logstash:										# 这两句非常重要，表示日志输出给logstash
  hosts: ["192.168.126.213:5044"]		# IP为logstash服务器的IP；端口5044对应logstash上的配置
  
processors:
  - add_host_metadata: ~
  - add_cloud_metadata: ~
```

第三步：启动或重启 filebeat

```yaml
filebeat# systemctl start filebeat
```

第四步：可以在浏览器中访问 Kibana 的 web 界面查看效果，也可以做成可视化的效果

> 以后我们就可以在每台业务服务器上安装 filebeat 去采集日志信息，将采集到的日志信息传递给 logstash，然后 logstash 进行过滤清晰将数据输出到 ES 中保存起来，最后使用 Kibana 去查询 ES 中的数据做成想要的可视化图表！
>
> 在业务系统上安装 filebeat 进行日志采集占用的内存比 logstash 要小很多。

## filebeat 收集 Nginx 日志（ELK 项目升级改造）

logstash 采集工作 => filebeat 采集

logstash 只负责日志处理，如 IP 解析、IP 过滤（127.0.0.1、本机）等等

第一步：在 filebeat 这台服务器上安装 Nginx，启动 Nginx 服务。并使用浏览器访问刷新一下，模拟产生一些相应的日志（强调：我们在这里是模拟的实验环境，一定要搞清楚，在实际情况下是将 filebeat 安装到 Nginx 所在的服务器上去收集日志的！）

```shell
filebeat# yum -y install nginx
filebeat# systemctl start nginx
filebeat# systemctl enable nginx
filebeat# echo 'filebeat server nginx' > /usr/share/nginx/html/index.html
```

第二步：修改 filebeat 配置文件，并重启服务

```yaml
filebeat# cat /etc/filebeat/filebeat.yml | grep -v '#' | grep -v '^$'
filebeat.inputs:
- type: log
  enabled: true
  paths:
    # 将Nginx日志路径单独指定，但需要排除以避免重复采集
    - /var/log/nginx/access.log
# 原始配置保持不变
filebeat.config.modules:
  path: ${path.config}/modules.d/*.yml
  reload.enabled: false

setup.template.settings:
  index.number_of_shards: 1
  
setup.kibana.enabled: false

output.logstash:
  hosts: ["192.168.126.213:5044"]

processors:
  - add_host_metadata: ~
  - add_cloud_metadata: ~
  
filebeat# systemctl stop filebeat
filebeat# systemctl start filebeat
```

第三步：验证是否在 ES 中有了数据（在 kibana 或 es-head 上查询），尽量使用 kibana 会显示更加详细一些。上面收集到的数据放在 ES 中索引是 filebeat2-xxxx.xx.xx 的了，因为我们本次没有修改 logstash 中的 xxx.conf 配置文件，最后一次使用的 xxx.conf 配置文件中定义了索引的名字！

> 注意：以上项目改造，省略了一个点，正常要采集 Nginx 的日志，需要把 Nginx 的日志转换为 json 格式，然后交给 logstash 进行处理。这个地方大家自行完成，详情参考 => ELK 项目实践

## 扩展：filebeat 日志简单过滤

filebeat 也可以进行日志或数据过滤操作，但是功能没有 logstash 那么强大：

https://www.elastic.co/guide/en/beats/filebeat/1.3/configuration-filebeat-options.html

```yaml
filebeat# grep -Ev '#|^$' /etc/filebeat/filebeat.yml

- type: log
  enabled: true
  paths:
    - /var/log/*.log

filebeat.config.modules:
  path: ${path.config}/modules.d/*.yml
  reload.enabled: false

setup.template.settings:
  index.number_of_shards: 1

setup.kibana.enabled: false

output.logstash:
  hosts: ["192.168.126.213:5044"]

processors:
  - add_host_metadata: ~
  - add_cloud_metadata: ~

```
