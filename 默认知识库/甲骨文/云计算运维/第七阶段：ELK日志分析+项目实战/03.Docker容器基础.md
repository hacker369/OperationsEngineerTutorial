# Paas 介绍

云计算 = 公有云 + 私有云 + 混合云 + 容器云

公有云：阿里云、华为云、百度云

私有云：OpenStack（底层）

混合云：公有云 + 私有云，比如 web 服务放在公有云，数据库放在私有云

容器云：Docker/Containerd + K8S

做云的优势：提高资源利用率，将资源打包做成服务给用户使用，资源提供商极少需要与用户交互打交道。我们可以直接购买云服务器，不需要自己建机房购买物理服务器，省心，另外很多云服务器已经将常用内容给提供好了。比如：Redis 集群，阿里云都是直接搭建好的，直接调用即可。

![1766492998703-2a2c9602-c778-4e53-96e6-cf256bc8d3ec.png](./img/is4zPIWMecFhuYBl/1766492998703-2a2c9602-c778-4e53-96e6-cf256bc8d3ec-908356.png)

Iaas（基础即服务）=> OpenStack（搭建云平台），阿里云、华为云，物理层、网络层 => 云平台基础层（底层）。简单来说就是厂商提供了平台，我们需要在平台上自己安装操作系统、部署软件、配置等等。

PaaS（系统即服务）=> 基础操作系统构建应用，搭建一个项目，需要一个软件一个软件的配置。简单来说就是厂商提供了操作系统，我们需要在操作系统上部署软件、配置等等。

SaaS（软件即服务）=> 用户不需要一个一个软件配置，提供了一套完整解决方案，只需要登录就可以直接使用。简单来说就是厂商提供了相应的软件，我们直接使用即可，也可以调整一些我们想要的配置。

FC 函数服务 => 主流大模型，如 DeepSeek。

***

由于 hypervisor 虚拟机技术仍然存在一些性能和资源使用效率方面的问题，因此出现了一种称为容器技术（Container）的新型虚拟化技术来帮助解决这些问题。

容器：是 PaaS 的一种实现，相对于虚拟机来说有更好的性能，更高的资源利用率。

在 IT 技术中：

虚拟化技术可以在宿主机上安装多个不同的操作系统，运行多套不同的应用。但可能就是为了运行一个 nginx，却还要在虚拟机里运行一个完整的操作系统、内核和其它无关程序，这种做法资源利用不高。

**所以我们希望更多的关注应用程序本身，而不再分精力去关注操作系统与无关程序，操作系统内核直接与宿主机共享。**

Linux 容器技术是一种**轻量级**的虚拟化技术。主要特点有：

* 轻量：只打包了需要的 bins/libs（也就是命令和库文件）。与宿主机共享操作系统，直接使用宿主机的内核。
* 部署快：容器的镜像相对虚拟机的镜像小。部署速度非常快，秒级部署。
* 移植性好：Build once，Run anywhere（一次构建，随处部署运行）。
* 资源利用率更高：相对于虚拟机，不需要安装操作系统，所以几乎没有额外的 CPU、内存消耗

面试题：Docker 容器与传统虚拟化（VM）之间有什么区别？

| 对比点 | Docker 容器 | 虚拟机（VM） |
| --- | --- | --- |
| 操作系统 | 与宿主机共享 OS | 宿主机 OS 上运行虚拟机 OS |
| 存储大小 | 镜像小，便于存储与传输 | 镜像庞大（vmdk、vdi 等） |
| 运行性能 | 几乎无额外性能损失 | 操作系统额外的 CPU、内存消耗 |
| 移植性 | 轻便、灵活，适用于 Linux | 笨重、与虚拟化技术耦合度高 |
| 硬件亲和性 | 面向软件开发者 | 面向硬件运维者 |
| 部署速度 | 快速、秒级 | 较慢，10s 以上 |

Docker 应用场景：部署各种各样应用，让我们实现秒级部署、迁移性比较强、性能几乎无额外损失！

## Docker 用到的内核技术

Docker A => Nginx

Docker B => Tomcat

相对独立，默认不会互相影响？

Docker 容器本质上是宿主机的**进程**，可以把 Docker 容器内部跑的进程看作是宿主机的线程。

Docker 通过 **namespace 实现了资源隔离**，通过 **cgroups 实现了资源限制**。

### NameSpace（了解）

Linux 内核实现 namespace 的一个主要目的就是实现轻量级虚拟化（容器）服务。在同一个 namespace 下的进程可以感知彼此的变化，而对外界的进程一无所知。

Linux 在很早的版本中就实现了部分的 namespace，比如内核 2.4 就实现了 mount namespace。

大多数的 namespace 支持是在内核 2.6 中完成的，比如 IPC、Network、PID 和 UTS。还有个别的 namespace 比较特殊，比如 User，从内核 2.6 就开始实现了，但在内核 3.8 中才宣布完成。

同时，随着 Linux 自身的发展以及容器技术持续发展带来的需求，也会有新的 namespace 被支持，比如在内核 4.6 中就添加了 Cgroup namespace。

Linux 内核提供了 6 种 namespace 隔离的系统调用。（了解）

| namespace | 系统调用参数 | 隔离内容 |
| --- | --- | --- |
| UTS | CLONE\_NEWUTS | 主机名或域名 |
| IPC | CLONE\_NEWIPC | 信号量、消息队列或共享内存 |
| PID | CLONE\_NEWPID | 进程编号 |
| net | CLONE\_NEWNET | 网络设备接口，IP 路由表、防火墙规则等 |
| mount | CLONE\_NEWNS | 挂载点（文件系统） |
| user | CLONE\_NEWUSER | 用户和用户组 |

UTS：每个 namespace 都拥有独立的主机名或域名，可以把每个 namespace 认为一个独立主机。

IPC：每个容器依旧使用 Linux 内核中进程交互的方法，实现进程间通信

PID：每个容器都拥有独立的进程树，而容器是物理机中的一个进程，所以容器中的进程是物理机的线程

Net：每个容器的网络是隔离的

Mount：每个容器的文件系统是独立的

User：每个容器的用户和组 ID 是隔离，每个容器都拥有 root 用户

小结：应用程序运行在一个隔离的空间（namespace）内，每个隔离的空间都拥有独立的 UTS、IPC、PID、Net、Mount、User。

### Control Group（了解）

控制组（Cgroups）是 Linux 内核的一个特性，主要用来对共享资源进行隔离、限制、审计等。

只有能控制分配到容器的资源，才能避免多个容器同时运行对宿主机系统的资源竞争。

控制组可以提供对容器的内存、CPU、磁盘 IO 等资源进行限制和计费管理。

### LXC 与 Docker 的区别

LXC 为 Linux Container 的简写。可以提供轻量级的虚拟化。

Docker 的底层就是使用 LXC 来实现的。Docker 以 LXC 为基础，实现了更多更强的功能。

资源隔离：namespace

资源限制：Cgroups

***

小结：

* Docker 容器属于 type III 虚拟化，属于 PaaS
* 容器是一种轻量级、进程级的虚拟机
* 相比于虚拟机的优势
  * 不需要安装 OS，和宿主机共享
  * 镜像存储空间小
  * 启动速度快（容器为秒级，虚拟机一般需要 10 秒左右）
  * 移植性更好、更轻便
  * 性能更好
* Docker 是一个实现容器的软件，底层使用 LXC
* Docker 主要使用 namespace 命名空间技术实现资源隔离，使用 cgroups 实现资源限制

宿主机：简单来说，就是运行了 Docker 程序的这个操作系统就是宿主机。

## Docker daemon 管理（可选，建议）

Docker 本身是 C/S 架构，拥有 Docker Client 与 Docker Server 服务器端。

Docker Client 与 Docker Server 抓紧的桥梁：docker daemon。

![1766589043349-11571989-5067-4feb-9c7c-a70e4aac55b2.png](./img/is4zPIWMecFhuYBl/1766589043349-11571989-5067-4feb-9c7c-a70e4aac55b2-787278.png)

可以将 client 与 server 进行分离，实现远程 docker 连接。为了实现它，就需要对 docker daemon 进行相应的配置。

```shell
# 查看信息
远程客户端主机# docker -H 容器宿主机IP version
```

配置过程

第一步：修改 Docker 配置文件前，请先关闭 Docker 守护进程

```shell
docker# systemctl stop docker

停止docker的时候会有警告：
Warning: Stopping docker.service, but it can still be activated by: docker.socket
意思是我们还需要再停止一下docker.socket

docker# systemctl stop docker.socket
```

第二步：通过 `/etc/docker/daemon.json`文件对 Docker 守护进程文件进行配置

```shell
docker# vim /etc/docker/daemon.json
{
  "hosts": ["tcp://0.0.0.0:2375","unix:///var/run/docker.sock"]
}
```

PS：docker daemon 默认侦听使用的是 unix 格式，侦听文件：unix:///run/docker.sock，添加 tcp://0.0.0.0:2375 可实现远程管理。

第三步：添加 `/etc/docker/daemon.json`后会导致 docker daemon 无法启动，请先修改如下文件内容：

```shell
修改前：
docker# vim /usr/lib/systemd/system/docker.service
[Service]
Type=notify
# the default is not to use systemd for cgroups because the delegate issues still
# exists and systemd currently does not support the cgroup feature set required
# for containers run by docker
ExecStart=/usr/bin/dockerd -H fd:// --containerd=/run/containerd/containerd.sock

修改后：
docker# vim /usr/lib/systemd/system/docker.service
[Service]
Type=notify
# the default is not to use systemd for cgroups because the delegate issues still
# exists and systemd currently does not support the cgroup feature set required
# for containers run by docker
ExecStart=/usr/bin/dockerd
```

第四步：修改完成后，一定要加载此配置文件

```shell
docker# systemctl daemon-reload
```

第五步：重新开启 docker 守护进程

```shell
docker# systemctl start docker
docker# ss -naltp | grep 2375
docker# ls /var/run/docker.sock
```

第六步：实例远程连接方法

```shell
远程客户端主机# docker -H 容器宿主机IP version

比如：
docker# docker -H 192.168.126.100 version
能够看到信息说明没问题
```

注意：客户端远程连接不需要加端口号

特别注意：远程客户端主机远程操作的权限非常大，建议请测试完后还原

小结：docker engine 分为 client 和 server，默认都在本地。（其实我们安装的 docker-ce 就是 docker engine）

# 四、镜像、容器与仓库

镜像（image）：镜像就是打包好的环境与应用。

容器（container）：容器就是运行镜像的实例。镜像看作是静态的，容器是动态的。

仓库（repository）：存放多个镜像的一个仓库。

> 镜像：类似日常生活中的光盘（静态）；容器：基于光盘播放的内容（动态）；仓库：相当于光盘专卖店。

Docker 近期由于未知原因关闭，给我们带来极大不便。为了解决这个问题，我们设置 Docker 镜像加速器。除此以外，我们还可以通过购买国外的云服务器来解决以上问题。（当然如果我们能够使用科学上网的话，也不需要进行任何配置）

```shell
docker# systemctl daemon-reload
docker# systemctl restart docker
```

## <font style="color:rgb(51, 51, 51);">镜像重命名</font>

<font style="color:rgb(51, 51, 51);">如果导入后看不到名称，可以使用</font><code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">docker tag</font></code><font style="color:rgb(51, 51, 51);">命令改名称 </font>

```shell
docker# docker images
REPOSITORY          TAG                 IMAGE ID            CREATED           SIZE
<none>              <none>              9f38484d220f        3 months ago      202 MB

docker# docker tag 9f38484d220f docker.io/centos:latest

docker# docker images
REPOSITORY          TAG                 IMAGE ID            CREATED          SIZE
docker.io/centos    latest              9f38484d220f        3 months ago      202 MB

语法：
docker# docker tag 镜像名称:版本 新镜像名称:版本
docker# docker tag 镜像ID 新镜像名称:版本
```

注意：docker tag 并不是把镜像复制一份，而是为其创建了一个链接，这个链接指向了原有镜像文件。

如果删除时，会有什么影响？

第一种情况：如果是通过镜像名称+tag 标签删除，则只删除引用，底层镜像并不会真正删除。

第二种情况：如果是通过镜像 ID 删除，则会清理所有引用关系，然后删除镜像本身。

问题：为什么容器运行完 hello-world 后就退出了，而不是继续运行？

我们前面把容器比喻为轻量级虚拟机，但是容器实际上只是进程。进程完事了当然就退出了，除非是类似服务那样的守护进程。

综上所述：容器大致可以分为两种类型：

一次性运行：如 hello-world，只运行 1 次就会自动结束（因为运行起来容器后，打印一句话就完事了）

永久性运行：如 Nginx、Tomcat、MySQL 容器，一旦启动后，就会一直运行

第三步：<font style="color:rgb(51, 51, 51);">使用另一台机器浏览器访问 </font>**<font style="color:rgb(51, 51, 51);">http://宿主机IP:8000 </font>**<font style="color:rgb(51, 51, 51);">测试</font>

![1767056920807-1180303b-637b-4eca-af10-547567f29cad.png](./img/is4zPIWMecFhuYBl/1767056920807-1180303b-637b-4eca-af10-547567f29cad-477917.png)

问题：`httpd -k start`是前台运行，为什么不能通过 `systemctl start httpd`后台运行？

```shell
答案：
Docker 的核心设计理念之一：一个容器 = 一个进程，而这个进程必须是前台运行的（Foreground），理由如下：
1. Docker 容器生命周期依赖于主进程
当我们运行一个 docker run 的时候，Docker 会创建一个容器，并运行你指定的主进程（比如：/bin/bash 或 httpd -D FOREGROUND）。
这个主进程一旦退出，容器就会停止运行。

2. 什么是前台和后台进程？
前台运行：进程在当前终端直接运行，不会 fork 到后台
示例：httpd -D FOREGROUND
后台运行（daemon 模式）：程序会 fork 自己到后台运行，然后主进程比如 /bin/bash 就没有任务要运行了，然后主进程退出。
示例：httpd 默认行为
在容器中，如果你用默认的后台方式启动服务，主进程退出 => Docker 认为容器结束 => 容器直接退出
```

### 案例 3：Docker 数据卷挂载（重点）

作用：保证 Docker 容器结束后，里面的数据不会随之消失，实现数据持久化存储。

问题：当我们有如下需求：

* 容器内配置文件需要修改
* 容器内数据（如：httpd 的家目录内的数据）需要保存
* 不同容器间数据需要共享（如：两个 httpd 容器家目录数据共享）

当容器删除时，里面的相关改变的数据也会删除，也就是说数据不能持久化保存。

我们可以将服务的配置文件、数据目录、日志目录等与宿主机的目录映射，把数据保持到宿主机上实现数据持久化。

宿主机的目录也可以共享给多个容器使用。

将宿主机的目录（数据卷）挂载到容器中（配置文件也可以挂载）

第一步：在宿主机创建一个目录，并建立一个内容不同的主页

```shell
docker# mkdir -p /docker_www
docker# echo lihupeng > /docker_www/index.html
```

第二步：运行容器 httpd4，将宿主机的 `/docker_www`目录挂载到容器中的 `/www/`目录

```shell
docker# docker run -d -p 8002:80 -v /docker_www/:/www --name=httpd4 httpd_image /usr/sbin/httpd -D FOREGROUND

docker# docker ps -l
CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS                  NAMES
484d7432d7ef        httpd_image         "/usr/sbin/httpd -..."          21 seconds ago      Up 20 seconds       0.0.0.0:8002->80/tcp   httpd4
```

第三步：使用另一台机器浏览器访问 http://宿主机 IP:8002 测试（注意：如果访问不到主页，请检查是否关闭了 SELinux）

第四步：尝试修改宿主机 `/docker_www/index.html`的内容，访问的结果也会随着修改而改变

***

默认情况下，Docker 中的数据卷挂载只能实现文件夹映射，但是如果只需要映射某个文件（情况很少），需要提前准备一个文件，然后再和 Docker 容器建立映射关系。

```shell
docker# mkdir -p /httpd_www /test
docker# yum -y install httpd
docker# cp /etc/httpd/conf/httpd.conf /test/httpd.conf

docker# docker run -d --name httpd5 -p 8000:80 \
> -v /httpd_www:/www \
> -v /test/httpd.conf:/etc/httpd/conf/httpd.conf \
> http_image /usr/sbin/httpd -D FOREGROUND
```

***

拓展：

如果我们进入之前使用 centos 镜像创建的容器中，查看当前的日期时间，发现是不对的，差 8 个小时

```shell

[root@82b985aea72c /]# date
```

默认的 centos 镜像时间与我们相差 8 小时，所以这是时区的差异，可以用如下方式解决：


# 九、使用 dockerfile 构建镜像（重点）

除了使用 `docker commit`把自定义容器提交成镜像外，还可以使用 dockerfile 来构建自定义镜像。

什么是 dockerfile？

dockerfile 把构建镜像的步骤都写出来，然后按照顺序执行实现自动构建镜像。就类似于脚本文件、Ansible 中的 playbook、saltstack 中的 sls 文件等。

## dockerfile 核心命令速查表

安装 docker => 编写 dockerfile => 构建最终镜像

| **<font style="color:rgb(51, 51, 51);">命令</font>** | **<font style="color:rgb(51, 51, 51);">作用</font>** | **<font style="color:rgb(51, 51, 51);">Nginx 典型示例</font>** |
| :--- | :--- | :--- |
| FROM | <font style="color:rgb(51, 51, 51);">指定基础镜像</font> | <code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">FROM nginx:1.25-alpine</font></code><font style="color:rgb(51, 51, 51);">（推荐轻量版）</font> |
| WORKDIR | <font style="color:rgb(51, 51, 51);">设置容器工作目录（非必须，但便于管理）</font> | <code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">WORKDIR /etc/nginx</font></code> |
| COPY | <font style="color:rgb(51, 51, 51);">复制本地文件到容器（覆盖默认配置或添加静态资源）</font> | <code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">COPY nginx.conf /etc/nginx/nginx.conf</font></code> |
| RUN | <font style="color:rgb(51, 51, 51);">执行命令（安装依赖、修改默认行为）</font> | <code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RUN yum install curl</font></code> |
| EXPOSE | <font style="color:rgb(51, 51, 51);">声明容器监听端口（需配合</font><code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">-p</font></code><font style="color:rgb(51, 51, 51);">映射到宿主机）</font> | <code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">EXPOSE 80 443</font></code> |
| VOLUME | <font style="color:rgb(51, 51, 51);">挂载数据卷（持久化日志或静态文件）</font> | <code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">VOLUME ["/var/log/nginx"]</font></code> |
| CMD | <font style="color:rgb(51, 51, 51);">启动 Nginx（通常不需修改，直接用基础镜像命令）</font> | <code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">CMD ["nginx", "-g", "daemon off;"]</font></code> |

> 设置了 workdir 就表示我们一进入容器中，就处于 workdir 设置的目录中了。

四句口诀：设环境，开端口，定命令，真灵巧。

```shell
定底像 => FROM
装程序 => RUN
拷文件 => COPY
选目录 => WORKDIR
设环境 => ENV
开端口 => EXPOSE
定命令 => CMD
真灵巧 => ENTRYPOINT
```

## <font style="color:rgb(51, 51, 51);">dockerfile 指令</font>

<font style="color:rgb(51, 51, 51);">通过</font><code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">man dockerfile</font></code><font style="color:rgb(51, 51, 51);">可以查看到详细的说明，我这里简单的翻译并列出常用的指令</font>

### <font style="color:rgb(51, 51, 51);">FROM</font>

<font style="color:rgb(51, 51, 51);">FROM 指令用于指定其后构建新镜像所使用的基础镜像。</font>

<font style="color:rgb(51, 51, 51);">FROM 指令必是 Dockerfile 文件中的首条命令。</font>

<font style="color:rgb(51, 51, 51);">FROM 指令指定的基础 image 可以是官方远程仓库中的，也可以位于本地仓库，优先本地仓库。</font>

```shell
格式:FROM <image>:<tag>
例:FROM centos:latest
```

### <font style="color:rgb(51, 51, 51);">RUN</font>

<font style="color:rgb(51, 51, 51);">RUN 指令用于在</font>**<font style="color:rgb(51, 51, 51);">构建</font>**<font style="color:rgb(51, 51, 51);">镜像中执行命令，有以下两种格式:</font>

<font style="color:rgb(51, 51, 51);">① shell 格式</font>

```shell
格式:RUN <命令>
例:RUN echo daniel > /var/www/html/index.html
```

<font style="color:rgb(51, 51, 51);">② exec 格式</font>

```shell
格式:RUN ["可执行文件", "参数1", "参数2"]
例:RUN ["/bin/bash", "-c", "echo daniel > /var/www/html/index.html"]
```

**<font style="color:rgb(51, 51, 51);">注意：</font>**<font style="color:rgb(51, 51, 51);">按优化的角度来讲，当有多条要执行的命令，不要使用多条 RUN，尽量使用 && 符号与 \ 符号连接成一行。因为多条 RUN 命令会让镜像建立多层(总之就是会变得臃肿了</font><font style="color:rgb(51, 51, 51);">)。</font>

```shell
RUN yum install httpd httpd-devel -y
RUN echo daniel > /var/www/html/index.html
可以改成
RUN yum install httpd httpd-devel -y && echo daniel > /var/www/html/index.html
或者改成
RUN yum install httpd httpd-devel -y  \
    && echo daniel > /var/www/html/index.html
```

### <font style="color:rgb(51, 51, 51);">CMD</font>

<font style="color:rgb(51, 51, 51);">CMD 不同于 RUN，CMD 用于指定在容器启动时所要执行的命令，而 RUN 用于指定镜像构建时所要执行的命令。</font>

```shell
格式有三种:
CMD ["executable","param1","param2"]
CMD ["param1","param2"]
CMD command param1 param2
```

<font style="color:rgb(51, 51, 51);">每个 Dockerfile 只能有一条 CMD 命令。如果指定了多条命令，只有最后一条会被执行。</font>

<font style="color:rgb(51, 51, 51);">如果用户启动容器时候指定了运行的命令，则会覆盖掉 CMD 指定的命令。</font>

```shell
什么是启动容器时指定运行的命令?
# docker run -d -p 80:80 镜像名 运行的命令
```

### <font style="color:rgb(51, 51, 51);">EXPOSE</font>

<font style="color:rgb(51, 51, 51);">EXPOSE 指令用于指定容器在运行时监听的端口。</font>

```shell
格式:EXPOSE <port> [<port>...]
例:EXPOSE 80 3306 8080
```

<font style="color:rgb(51, 51, 51);">上述运行的端口还需要使用 docker run 运行容器时通过 -p 参数映射到宿主机的端口。</font>

### <font style="color:rgb(51, 51, 51);">ENV</font>

<font style="color:rgb(51, 51, 51);">ENV 指令用于指定一个环境变量。</font>

```shell
格式:ENV <key> <value> 或者 ENV <key>=<value>
例:ENV JAVA_HOME /usr/local/jdkxxxx/
```

### <font style="color:rgb(51, 51, 51);">ADD</font>

<font style="color:rgb(51, 51, 51);">ADD 指令用于把宿主机上的文件拷贝到镜像中。</font>

```shell
格式:ADD <src> <dest>
<src>可以是一个本地文件或本地压缩文件，还可以是一个url,
如果把<src>写成一个url，那么ADD就类似于wget命令
<dest>路径的填写可以是容器内的绝对路径，也可以是相对于工作目录的相对路径
```

### <font style="color:rgb(51, 51, 51);">COPY</font>

<font style="color:rgb(51, 51, 51);">COPY 指令与 ADD 指令类似，但 COPY 的源文件只能是本地文件（文件或目录）</font>

```shell
格式:COPY <src> <dest>
```

<font style="color:rgb(51, 51, 51);">ADD 与 COPY 最佳实践：</font>

<font style="color:rgb(51, 51, 51);">默认优先使用 COPY</font>

<font style="color:rgb(51, 51, 51);">仅当需要解压本地压缩包时使用</font>

```shell
ADD app.tar.gz /app/
等价于
COPY app.tar.gz /app/ && tar -xzf /app/app.tar.gz -C /app/
```

### <font style="color:rgb(51, 51, 51);">ENTRYPOINT</font>

<font style="color:rgb(51, 51, 51);">ENTRYPOINT 与 CMD 非常类似</font>

<font style="color:rgb(51, 51, 51);">相同点：一个 Dockerfile 只写一条，如果写了多条，那么只有最后一条生效都是容器启动时才运行</font>

<font style="color:rgb(51, 51, 51);">不同点：如果用户启动容器时候指定了运行的命令，ENTRYPOINT 不会被运行的命令覆盖，而 CMD 则会被覆盖</font>

```shell
格式有两种:
ENTRYPOINT ["executable", "param1", "param2"]
ENTRYPOINT command param1 param2
```

### <font style="color:rgb(51, 51, 51);">VOLUME</font>

<font style="color:rgb(51, 51, 51);">VOLUME 指令用于把宿主机里的目录与容器里的目录映射。</font>

<font style="color:rgb(51, 51, 51);">只指定挂载点，docker 宿主机映射的目录为自动生成的。</font>

```shell
格式:VOLUME ["<mountpoint>"]
```

### <font style="color:rgb(51, 51, 51);">USER</font>

<font style="color:rgb(51, 51, 51);">USER 指令设置启动容器的用户(像 hadoop 需要 hadoop 用户操作，oracle 需要 oracle 用户操作)，可以是用户名或 UID。</font>

```shell
USER daemon
USER 1001
```

**<font style="color:rgb(51, 51, 51);">注意</font>**<font style="color:rgb(51, 51, 51);">：如果设置了容器以 daemon 用户去运行，那么 RUN、CMD 和 ENTRYPOINT 都会以这个用户去运行镜像构建完成后，通过docker run 运行容器时，可以通过 -u 参数来覆盖所指定的用户。</font>

### <font style="color:rgb(51, 51, 51);">WORKDIR</font>

<font style="color:rgb(51, 51, 51);">WORKDIR 指令设置工作目录，类似于 cd 命令。不建议使用</font><code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RUN cd /root</font></code><font style="color:rgb(51, 51, 51);">，建议使用 WORKDIR。</font>

```shell
WORKDIR /root
```

***

使用 dockerfile 构建镜像的步骤：

1. 创建一个文件夹（目录）
2. 在文件夹（目录）中创建 dockerfile 文件（并编写）及其他文件
3. 使用 `docker build`命令构建镜像
4. 使用构建的镜像启动容器

## <font style="color:rgb(51, 51, 51);">案例 1：Dockerfile 构建 httpd 镜像 v1</font>

第一步：<font style="color:rgb(51, 51, 51);">准备一个目录(自定义)</font>


## Dockerfile 最佳实践（获取一个最小镜像）

要求：此规则最少要记住 4-5 条左右，面试常考内容。

<font style="color:#C75C00;">① 构建层最小化：合并 RUN 指令减少镜像层数（也就是将多条 RUN 指令合成一条去写）</font>

<font style="color:#C75C00;">② 缓存优化：将变化少的指令放在前面</font>

当你改动 Dockerfile 里某一条命令时，从这一层往下的所有层都会重新构建，而前面的没变的层直接用之前的缓存，不再执行一次。

所以，把很少变的命令放在前面，容器变动的命令放在后面，这样一改代码，只有后面的几层会重新 build，前面的不用浪费时间。

不佳写法：

```shell
FROM node:20.11-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "app.js"]

注意：这样，每次你哪怕只改动了一行js代码，COPY . . 这一行就会导致缓存失效，npm install 每次都要重新跑，非常慢。
```

好的写法：

```shell
FROM node:20.11-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
CMD ["node", "app.js"]
```

<font style="color:#C75C00;">③ 最小安装：只安装必要的包和依赖（可以选择那种轻量级的镜像，比如 alpine 的）</font>

<font style="color:#C75C00;">④ 使用特定标签：避免使用 latest 标签（因为 latest 是最新的，往往比较大，可以指定我们使用的镜像版本，比如 httpd:2.4 等）</font>

<font style="color:#C75C00;">⑤ 多阶段构建：分离构建环境和运行环境</font>

单阶段

```shell
FROM python:3.11
WORKDIR /app
COPY . .
RUN pip install --no-cache-dir -r requirements.txt
CMD ["python", "app.py"]

缺点：镜像里可能包含了pip缓存、临时编译文件、.pyc文件、开发工具、测试依赖等，体积大、不安全。
```

多阶段：

```shell
# 第一阶段，只做依赖和编译，只留下真正需要的包，避免开发/测试依赖
FROM python:3.11-slim AS builder
WORKDIR /install
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install/deps -r requirements.txt

# 第二阶段，极简运行环境，只COPY刚才安装好的依赖和你的应用代码，没有pip缓存，没有编译、构建时的中间垃圾文件，极小、极干净
FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /install/deps /usr/local
COPY . .
CMD ["python", "app.py"]
```

<font style="color:#C75C00;">⑥ 非 root 用户：使用最小权限原则</font>

本质上：非 root 用户不是直接减少体积的原因，而是极简、非 root 习惯，常常配套极小的基础镜像，一起用，体积更小。

<font style="color:#C75C00;">⑦ 合理排序：依赖安装在代码复制前</font>

Docker 在构建镜像时，每一条指令都会创建一个镜像层，如果某一层发生变动，这一层后面的所有层都会重新构建，但这一层之前的可以直接用缓存。

不佳写法：

```shell
COPY . .
RUN pip install -r requirements.txt

你只要改一行代码，COPY . . 这一层的缓存就失效了，导致每次都要重新执行 pip install （或 npm install），很慢！
```

推荐写法：

```shell
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .

只要你的依赖（requirements.txt）没变，pip install 这一层就用缓存，你改代码，只会影响 COPY . . 这一层，依赖不需要重装，构建速度极快！
```

<font style="color:#C75C00;">⑧ 使用 .dockerignore：排除不需要的文件</font>

如：.git、node\_modules、文档等，加快构建速度、减少镜像体积。

```shell
__pycache__
*.pyc
*.pyo
*.pyd
.git
tests/
.env
Dockerfile
*.log
```

<font style="color:#C75C00;">⑨ 注意安全更新：定期更新基础镜像</font>

关注官方安全公告，及时 rebuild。

# 十、Docker 容器资源限制（记住）

在宿主机中我们可能会启动很多个容器，所以需要避免某些容器占用过多的资源导致别的容器用不了。

## 内存限制

\--memory，-m：限制最大内存（单位：b、k、m、g）

```shell
# 只允许使用512M内存
docker run -m 512m nginx
```

\--memory-swap：限制内存+swap 的总和

```shell
# 内存512M，总共1G（含swap）
docker run -m 512m --memory-swap 1g nginx
```

\--memory-reservation：软限制，超过此值才被回收

```shell
docker run --memory-reservation=256m -m 512m nginx

内存达到512M后，并且如果在512M的基础上再次使用超过256M后进行回收超出的部分
```

## CPU 限制

\--cpus：限制最多可用的 CPU 核数（支持小数）

```shell
# 最多使用0.5核
docker run --cpus=0.5 nginx
```

\--cpu-shares：CPU 权重，默认 1024，值越大优先级越高

\--cpu-shares 并不是直接限定某容器可用的 CPU 数量，而是用于在 CPU 资源竞争时分配“权重”，默认值是 1024，支持设置为更大或更小的正整数。

CPU 空闲时：任何容器（进程）都可以尽可能多地使用可用 CPU，不受 cpu-shares 影响。

CPU 紧张/多容器竞争时：CPU 时间按照各自 cpu-shares 的权重分配，例如：

A 容器分配了 1024

B 容器分配了 512

在 CPU 资源紧张时，A 理论上将获得 2/3 的 CPU 时间，B 获得 1/3

```shell
docker run --cpu-shares=512 nginx
```

\--cpuset-cpus：指定在哪些物理核心上运行

```shell
# 仅用第1和第2号CPU
docker run --cpuset-cpus="1,2" nginx
```

## 磁盘 IO 限制

需要宿主机支持 cgroup blkio。

\--device-read-bps/--device-write-bps：限制每秒读/写速率

```shell
# 限制最大每秒1MB读速率
docker run --device-read-bps /dev/sda:1mb nginx
```

\--device-read-iops/--device-write-iops：限制每秒 IO 次数

```shell
docker run --device-write-iops /dev/sda:100 nginx
```

## 进程/文件句柄数限制

\--ulimit：限制容器内可用的系统资源

```shell
# 最大进程数128，最大打开文件数1024
docker run --ulimit nproc=128 --ulimit nofile=1024:1024 nginx
```

文件句柄数（file descriptor，FD），又叫“打开文件数”或“nofile”，指的是一个进程（或整个系统）同时能打开的最大文件数/网络连接数/管道数等资源。

它不仅仅是“文件”，包括所有类型的 I/O 对象，比如：

* 普通文件
* 目录
* Socket（网络连接）
* 管道（pipe）
* 设备（比如串口）

Linux 下，每打开一个文件、网络连接或管道，都会分配一个唯一的文件描述符。如果超出上限，新文件或网络连接就会打开失败，报错“Too many open files”，服务会异常，严重时业务崩溃。

生产环境怎么设置文件句柄数？

```shell
# 查看系统级别上限
ulimit -n

# 查看全局最大上限
cat /proc/sys/fs/file-max

# 临时提升当前会话（仅当前shell）
ulimit -n 65536

# 永久提升（系统级），编辑/etc/security/limits.conf，添加
* soft nofile 65536
* hard nofile 65536
注：重启或重新登录生效
```

## 组合案例

```shell
docker run -d \
  --name test-nginx \
  -m 256m \
  --cpus=0.5 \
  --cpuset-cpus="0,1" \
  --ulimit nofile=65536:65536 \
  nginx
```

> 上述命令：最大内存 256M、最多 0.5 核、限定 CPU 使用 0 和 1 号核心、最大打开文件数 65536。

## 查看资源限制和实时用量

实时查看资源使用情况

```shell
docker stats
```

查看具体容器资源限制参数

```shell
docker inspect 容器ID/容器名
```

## 超过资源限制会发生什么

内存超限：

* 容器会被直接 kill（OOM），容器内进程会收到 `Out of Memory`错误，部分进程可能被强制终止。
* 具体表现：容器重启或退出（取决于 restart 策略），可用 `docker logs`查看错误。

CPU 超限：

* 容器进程会被限速，不会被 kill，只是运行变慢

IO 限制：

* 达到速率或次数上限后，IO 操作变慢

ulimit 超限：

* 超过进程数或文件句柄数时，容器内新进程无法创建，或者文件无法打开，会报错

## 注意事项

* 没有设置资源限制时，容器可占满主机所有资源（极端情况下会拖垮主机）
* 资源限制参数只影响本容器，其他容器不受影响
* 资源限制可以组合使用
* 磁盘空间限制可以通过卷管理或宿主机 quota 工具实现
