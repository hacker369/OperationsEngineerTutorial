# 01.Containerd容器运行时

# 一、【了解】Containerd 发展历史

**为什么要学它？**

* \*\*核心地位：\*\*Containerd 处于容器技术栈的基础层面，负责容器的生命周期管理，包括容器的创建、启动、停止、销毁等操作。
* **技术基石**：它是理解更高级容器编排工具（如 Kubernetes）的基础，了解 Containerd 有助于你更好地理解 Kubernetes 是如何与底层容器进行交互的。

## Containerd 的前世今生

> 云计算背景
>
> 2006年，亚马逊公司宣布推出AWS（Amazon Web Service）平台。次年又推出了在线存储服务S3（Simple Storage Service）和弹性计算云EC2（Elastic Compute Cloud）等服务。但当年亚马逊并没有谷歌和IBM的影响力，并未引起业界震动。后来随着云计算的爆发，亚马逊在商业模式上的积累，奠定了AWS的高速发展和领军地位。
>
> 2006年，谷歌推出了“Google 101计划”，并正式提出“云”的概念和理论。该计划基于谷歌员工比希利亚的设想，初衷是设置一门课程，着重引导学生们进行“云”系统的程序开发;
>
> 2007年10月，谷歌、IBM联合了美国6所知名大学帮助学生在大型分布式计算系统上进行开发，当时的IBM发言人就指出这种所谓的“大型分布式计算系统”就是云计算，明确提出“云计算”这一新概念。
>
> 2008年10月，微软公司的Windows Azure Platform公共云计算平台发布，开始了微软公司的云计算之路。
>
> 2010年7月，美国太空总署联合Rackspace、AMD、Intel、戴尔等厂商共同宣布“OpenStack”开源计划。成立了**OpenStack开源云平台**。

\*\*【初次邀约】\*\*2013年Docker公司在推出docker产品后,由于其对全球技术产生了一定的影响力，Google公司明显感觉到自己的江湖地位受到了威胁，希望Docker公司能够与自己联合打造一款开源的容器运行时作为Docker核心依赖，但Docker公司拒绝了。

**【成立开发容器倡议社区】**接着Google公司联合RedHat、IBM等公司说服Docker公司把其容器核心技术**libcontainer**捐给中立社区\*\*(OCI,Open Container Intiative)**，并更名为**runC\*\*。

**【成立云原生基金会】2015年，**为了进一步遏制Docker在未来技术市场影响力，避免在容器市场上Docker一家独大，Google公司带领导RedHat、IBM等成立了<font style="color:rgb(216,57,49);">CNCF(Cloud Native Computing Fundation)基金会</font>，即**云原生**计算基金会。

\*\*【甩出秘密武器】\*\*CNCF的目标很明确，既然在容器应用领域无法与Docker相抗衡，那就做Google更有经验的技术市场------大规模容器编排应用场景，Google公司把自己内部使用的Borg系统开源------Kubernetes，也就是我们今天所说的云原生技术生态。

> 容器编排是一项能够完成容器部署、调度、伸缩以及健康监控等（全生命周期）任务的自动化管理技术。

> Borg
>
> Borg 是《星际迷航》虚构的一个宇宙种族——博格人。每个博格人没有什么自主意识，整体受一个Borg Queen控制，集群拥有高度智慧和战斗能力。
>
> 受电影启发，谷歌内部用Borg命名公司的集群管理系统。

2015年Docker公司推出了Docker Swarm，意在一统Docker生态，让Docker既可以实现容器应用管理，也可以实现大规模容器编排，经过近1年左右时间的市场验证后，发现在容器编排方面无法独立抗衡Kubernetes，所以Docker公司于2017年正式宣布原生支持Kubernetes。

至此，Docker在大规模容器编排应用市场败下阵来。但是Docker依然不甘心失败，把Docker核心依赖Containerd捐给了CNCF，以此说明Docker依旧是一个PaaS平台（目的是为了**在容器生态中占有一席之地**）。

2020年CNCF基金会宣布Kubernetes 1.20版本将不再仅支持Docker容器管理工具，所以接下来Kubernetes容器运行时的主角是Containerd。涵盖整个**容器运行时**管理的所有需求。

> K8s 1.24 版本切换成了 Containerd。
>
> 补充：运行时(Runtime)，在计算机中，是指程序的运行阶段、运行所依赖的环境、在运行过程中可能出现的各种情况。

## dnf 安装

### 获取阿里源

在 CentOS Stream 9 系统中，我们可以从阿里云获取适用于该系统的 Docker DNF 源，该源包含了 Containerd 软件的相关信息，便于后续安装。

```shell
# 先查看当前的镜像源

# 如果不是阿里源，再执行以下命令，从阿里云下载 Docker DNF 源配置文件到系统指定目录

dnf install -y wget

# -O 输出到指定文件
wget -O /etc/yum.repos.d/docker-ce.repo https://mirrors.aliyun.com/docker-ce/linux/centos/docker-ce.repo
```

## 帮助手册

不同环境下 Docker、Containerd 及 Kubernetes 管理镜像的命令工具解析：

* Docker 使用 docker images 命令管理镜像。
* 在单机环境下，Containerd 使用 ctr images 命令管理镜像，其中 ctr 是 Containerd 本身自带的命令行接口（CLI）工具。
* 在 Kubernetes（K8s）集群中，若使用 Containerd 作为容器运行时，可使用 crictl images 命令管理镜像，crictl 是 Kubernetes 社区开发的用于与容器运行时交互的专用命令行接口（CLI）工具。

# 四、【掌握】Containerd 容器管理

## 帮助手册

\*\*镜像（Image）：\*\*镜像是一个只读的模板，它包含了运行应用程序所需的所有文件、依赖项、环境变量和配置信息等。类似于软件安装包；

\*\*容器（Container）：\*\*容器是镜像的运行时实例。当使用镜像创建一个容器时，会在镜像的基础上添加一个可写的层，用于存储容器运行过程中产生的数据和状态。只有当任务启动后，才会开始占用系统资源。

\*\*任务（Task）：\*\*任务是容器内具体的执行单元，它代表了容器内正在运行的进程。

命令 `ctr --help` 和命令 ctr 是等价的。

```shell
ctr
```

### 修改 Containerd 配置

> 说明：config\_path 是 containerd 1.5 版本后，新加的功能，意思是从指定目录下，自动去加载对应镜像的配置文件。

在**安装 containerd 的主机**上，依次执行：

第一步：添加 config.toml 中的 \[plugins.cri.registry] 选项

```shell
# 生成配置文件
[root@container ~]# containerd config default > /etc/containerd/config.toml

# 此配置文件已提前替换过，仅修改本地容器镜像仓库地址即可。
[root@container ~]# vim /etc/containerd/config.toml

# config_path 改成以下的内容
[root@container ~]# sed -i 's/config_path = ""/config_path = "\/etc\/containerd\/certs.d\/"/g' /etc/containerd/config.toml

# 最后效果是下面这样
[plugins.cri.registry]
  config_path = "/etc/containerd/certs.d"
```

第二步：在 /etc/containerd/ 目录下，创建一个 certs.d 目录

```shell
[root@container ~]# cd /etc/containerd
[root@container containerd]# mkdir -p certs.d/harbor.lhp.cn  # 域名改成自己配置的harbor服务器的域名
[root@container containerd]# touch /etc/containerd/certs.d/harbor.lhp.cn/hosts.toml 
```

第三步：执行以下命令

```shell
[root@container containerd]# vi /etc/containerd/certs.d/harbor.lhp.cn/hosts.toml

# ------- 粘贴进去以下内容，保存退出 ---------------
server = "http://harbor.lhp.cn"		# 域名改成自己配置的Harbor服务器的域名

[host."http://harbor.lhp.cn"]			# 域名改成自己配置的Harbor服务器的域名
  capabilities = ["pull", "resolve"]
```

### ctr 上传镜像

上传到 Harbor 的 library 公有项目中。

```shell
# 重新生成新的tag

# 如果之前拉取过，这两个本地应该都有
```

```shell

```

```shell
# 推送容器镜像至Harbor，
ctr images push --platform linux/amd64 --plain-http -u admin:123 harbor.lhp.cn/library/nginx:1.14.1-8.6

其实建议当时拉取到时候平台指定的是什么，推送的时候还得是什么。比如之前是Linux的平台，那就还用Linux的平台；之前是全平台，那就还用全平台。
```

说明：

1、推镜像的时候，要先 tag 再 push；

2、因为我们 harbor 是 http 协议，不是 https 协议，所以<font style="color:rgb(216,57,49);">一定要加上 --plain-http</font>；

3、--user admin:123 是指定 harbor 的用户名与密码。密码改成自己的；

### ctr 下载镜像

在 Harbor 仓库中我们目前有两个公开的项目 library 和 beijing。

公开的项目表示：下载镜像不需要登录，但是上传镜像时需要登录的；

私有的项目表示：下载镜像和上传镜像都需要登录。

```shell
# 先把本地的删掉一下
[root@containerd ~]# ctr i rm harbor.lhp.cn/library/nginx:1.14.1-8.6

[root@containerd ~]# ctr images ls

# 下载容器镜像，pull表示拉取
# --plain-http 指的是通过 http 协议，即不走443端口，而是走80端口
[root@containerd ~]# ctr images pull --plain-http harbor.lhp.cn/library/nginx:1.14.1-8.6

# 查看已下载容器镜像
[root@containerd ~]# ctr images ls
```

# 七、【课下作业】二进制安装

Containerd 有两种安装包：

* 第一种是containerd-xxx，这种包用于单机测试没问题，不包含runC，需要提前安装。
* 第二种是cri-containerd-cni-xxxx，包含runc和k8s里的所需要的相关文件。k8s集群里需要用到此包。虽然包含runC，但是依赖系统中的seccomp（安全计算模式，是一种限制容器调用系统资源的模式。）

### 生成 containerd 模块配置文件

1 生成默认模块配置文件

Containerd 的默认配置文件为 /etc/containerd/config.toml，可以使用containerd config default > /etc/containerd/config.toml命令创建一份模块配置文件

```shell
# 创建配置文件目录
mkdir /etc/containerd

# 生成配置文件
containerd config default > /etc/containerd/config.toml

# 查看配置文件
cat /etc/containerd/config.toml
```

![1767660437903-07f589a6-185b-4a50-9f82-349bf34e9e0d.png](./img/_g0UN1i-r3VtrgBF/1767660437903-07f589a6-185b-4a50-9f82-349bf34e9e0d-426092.png)

2 替换默认配置文件（可选）

但上述配置文件后期改动的地方较多，这里直接换成可单机使用也可k8s环境使用的配置文件并配置好镜像加速器。

```shell
vim /etc/containerd/config.toml

# 把下面的内容粘贴进去
root = "/var/lib/containerd"
state = "/run/containerd"
oom_score = -999

[grpc]
  address = "/run/containerd/containerd.sock"
  uid = 0
  gid = 0
  max_recv_message_size = 16777216
  max_send_message_size = 16777216

[debug]
  address = ""
  uid = 0
  gid = 0
  level = ""

[metrics]
  address = ""
  grpc_histogram = false

[cgroup]
  path = ""

[plugins]
  [plugins.cgroups]
    no_prometheus = false
  [plugins.cri]
    stream_server_address = "127.0.0.1"
    stream_server_port = "0"
    enable_selinux = false
    sandbox_image = "easzlab/pause-amd64:3.2"
    stats_collect_period = 10
    systemd_cgroup = false
    enable_tls_streaming = false
    max_container_log_line_size = 16384
    [plugins.cri.containerd]
      snapshotter = "overlayfs"
      no_pivot = false
      [plugins.cri.containerd.untrusted_workload_runtime]
        runtime_type = ""
        runtime_engine = ""
        runtime_root = ""
    [plugins.cri.cni]
      bin_dir = "/opt/kube/bin"
      conf_dir = "/etc/cni/net.d"
      conf_template = "/etc/cni/net.d/10-default.conf"
    [plugins.cri.x509_key_pair_streaming]
      tls_cert_file = ""
      tls_key_file = ""
  [plugins.diff-service]
    default = ["walking"]
  [plugins.linux]
    shim = "containerd-shim"
    runtime = "runc"
    runtime_root = ""
    no_shim = false
    shim_debug = false
  [plugins.opt]
    path = "/opt/containerd"
  [plugins.restart]
    interval = "10s"
  [plugins.scheduler]
    pause_threshold = 0.02
    deletion_threshold = 0
    mutation_threshold = 100
    schedule_delay = "0s"
    startup_delay = "100ms"
```
