## 信号参数

kill 命令的作用是传输信号给进程

> kill 不仅仅可以杀死进程，其底层核心是给进程传递一个信号！因为 kill -9 与 kill -15 使用较多，给大家带来错觉！

```shell
-TERM，-INT	(快速退出，当前的请求不执行完成就退出)
-QUIT				(优雅退出，执行完当前的请求后退出)
-HUP				(重新加载配置文件，用新的配置文件启动新worker进程，并优雅的关闭旧的worker进程)
-USR1				(重新打开日志文件)
-USR2				(平滑的升级nginx二进制文件，拉起一个新的主进程旧主进程不停止)
-WINCH			(优雅的关闭worker进程)


重点:
平滑升级
kill -USR2 pid
关闭master下面的所有work(子进程)
kill -WINCH pid
优雅的退出master
kill -QUIT pid
```

> 使用 kill -l 可以查看 kill 命令可以传递的所有信号

## <font style="color:rgb(51, 51, 51);">平滑升级（重点）</font>

虽然通过上面的方式可以实现 Nginx 的升级，但是如果直接把旧版本的服务停止掉，会影响线上业务的使用。那我们该如何进行升级呢?

答：可以采用平滑升级(不停服升级)，大致可以分为以下三个步骤

```shell
1. 旧的先不停掉
2. 新的又可以起来
3. 旧的和新的同时提供服务，旧的请求完成之后，就停掉旧进程
-USR2 	平滑启动一个新的进程(平滑升级)
-WINOH 	优雅的关闭子进程（Nginx分为master主进程和worker子进程）
-QUIT 	优雅关闭旧的nginx的主进程

启动Nginx后，会产生两个进程，一个主进程master，一个是子进程worker！
```

小结:

工作中经常需要对旧版本程序进行升级操作，如 MySQL、Nginx 等等，但是 Nginx 本身有两种升级方式。

重装升级：需要停服，然后删原目录，重新编译安装!

平滑升级：不需要停服，直接在原有基础上进行升级!

`kill -USR2`平滑升级

`kill -WINCH`移除当前进程的所有子进程(worker 进程)

`kill -QUIT`代表优雅的关闭 master 主进程

# 二、企业级服务配置

## nginx.conf 配置文件

nginx.conf 存储位置有两种情况：① dnf/yum 安装的，则通常在`/etc/nginx/nginx.conf`目录下 ② 自定义安装，大多数都是位于 Nginx 软件安装目录

编译的 Nginx 默认的配置文件位置：`/usr/local/nginx/conf/nginx.conf`

使用 grep 删除注释与空行：`grep -Ev '#|^$' conf/nginx.conf > conf/nginx.conf`

```shell
# cd /usr/local/nginx
# grep -Ev '#|^$' conf/nginx.conf
```

![1764250019211-3c978309-93c2-4879-8b0a-bb8e212ea2a0.png](./img/Q565A6Gwp4Fr9oxm/1764250019211-3c978309-93c2-4879-8b0a-bb8e212ea2a0-535077.png)

说明：

**worker\_processes**：它的值越大，能够并发处理的请求就越多，这个值最高可以和 CPU 的核心数保持一致，比如 2 核心 CPU，那值最高就是 2，4 核心的 CPU，那值最高就是 4。

**error\_log**：表示错误日志的存放位置

**pid**：Linux 中每个进程都是一个文件，这里表示 Nginx 进程所在的文件位置

**worker\_connections**：表示每个 worker\_process 能够处理的并发请求数，如果 worker\_processes 的值是 4，那么表示能够并发处理的请求可以达到 4096 个。

**http**：每个 nginx.conf 只能有一个 http 模块，http 模块中可以有多个 server 模块，每个 server 模块中可以有多个 location 模块。

**include**：表示 Nginx 中处理的数据类型

**default\_type**：表示默认的数据类型，上图中默认是二进制流数据类型，表示文件、字符串等都能处理。

**keepalive\_time**：超时时间，表示本次连接 65 秒都没有任何动作，就会关闭该连接

**server**：每个项目就对应一个 server 模块（其实也就是类似于 httpd 服务器中学习的虚拟主机的概念）

**listen**：表示监听的端口

**server\_name**：服务器名称，表示域名地址，以后如果申请了域名，那么域名就写在这里

**location**：一个 server 中可以有多个 location，表示匹配请求的 url 地址，上图中 / 表示匹配所有请求，`location = /50x.html`表示匹配 url 路径是 /50x.html 的请求

重点：学会配置 server 区块

```nginx
server {
    listen       80;
    server_name  localhost;
    root				 html;
    location / {
        index  index.html index.htm index.php;
    }
    location ~ \.php$ {
            fastcgi_pass   127.0.0.1:9000;
            fastcgi_index  index.php;
            fastcgi_param  SCRIPT_FILENAME  $document_root$fastcgi_script_name;
            include        fastcgi_params;
    }
    error_page   500 502 503 504  /50x.html;
    location = /50x.html {
    }
}

listen：监听端口，默认为80
server_name：很重要，可以绑定域名(更多一些)、可以绑定IP，localhost本机虚拟域名，底层IP地址就是127.0.0.1
root html：代表Nginx项目目录，从安装目录开始算起
location：url匹配规则，location = /50x.html精准匹配，location ~ \.php$正则匹配，location / 代表如果刚才那些匹配规则都没有匹配上，则默认匹配location / 
error_page 500 502 503 504 /50x.html：如果服务器端响应500、502、503、504，则自动跳转到/50x.html这个错误页面
fastcgi_pass：请求转化给后面的地址
fastcgi_index：动态请求中的默认首页
fastcgi_param：参数设计，SCRIPT_FILENAME代表脚本名称。$document_root代表root指定的项目目录，$fastcgi_script_name请求的文件名称。
```

## server 区块配置（虚拟主机）

### 基于域名虚拟主机（重要）

在实际生产业务环境中，一台 web 服务器，需要使用多个网站部署。搭建 vhost 虚拟机主机实现不同域名，解析绑定到不同的目录。

www.shop.com => /usr/local/nginx/html/shop

www.devops.com => /usr/local/nginx/html/devops

案例：添加一个 www.devops.com 域名，解析到 `/usr/local/nginx/html/devops`项目目录

第一步：编辑 nginx.conf 文件，添加虚拟主机

```nginx
# vim /usr/local/nginx/conf/nginx.conf
# 添加虚拟主机
server {
    # 监听端口
    listen 80;
    # 配置域名
    server_name www.devops.com;
    # 配置目录
    root html/devops;
    # 配置url匹配规则
    location / {
        # 默认索引页
        index index.html index.htm;
    }
}


# 检测配置文件是否有问题
# /usr/local/nginx/sbin/nginx -t
nginx: the configuration file /usr/local/nginx/conf/nginx.conf syntax is ok
nginx: configuration file /usr/local/nginx/conf/nginx.conf test is successful
```

第二步：在 Windows 电脑中，解析 hosts 域名，设置 IP 与域名映射，强制让 DNS 把 www.devops.com 指向虚拟机 IP

\*\*问题：\*\*如果在 nginx.conf 配置文件中，只绑定了 192.168.126.200 这个 IP，没有绑定 192.168.126.174 这个 IP，那么浏览器中访问 192.168.126.174 时会转向哪里？

\*\*答：\*\*如果 IP 地址与配置文件中的 server 区块相匹配（也就是有对应的 server\_name），则直接显示匹配的信息。如果 IP 地址与配置文件中的 server 区块都不匹配，则会默认指向第一个 server 区块！

\*\*比如：\*\*上面的 Nginx 配置文件中我们的虚拟主机一个是 localhost、一个是 192.168.126.200。就没有配置与 192.168.126.174 对应的区块，那就会匹配第一个配置好的 server 区块！

## GZIP 压缩（优化手段）

### 概述

压缩文件大小变小了，传输更快了。目前市场上大部分浏览器是支持 GZIP 的。IE6 以下支持不好，会出现乱码情况。

<font style="color:#117CEE;">http://nginx.org/en/docs/http/ngx\_http\_gzip\_module.html</font>

```shell
gzip on;
gzip_min_length 1k;
gzip_buffers 4 16k;
gzip_http_version 1.1;
gzip_comp_level 5;
gzip_types text/plain text/css text/javascript application/x-javascript image/jpeg image/gif image/png image/x-ms-bmp;
gzip_vary on;
gzip_disable "MSIE [1-6]\.";

第1行:开启Gzip
第2行:不压缩临界值，大于1K的才压缩，一般不用改
第3行:gzip buffers number,size，压缩文件时使用的缓存空间的大小，默认128
第4行:用了反向代理的话，末端通信是HTTP/1.0，默认是HTTP/1.1
第5行:压缩级别，1-10，数字越小压缩的越好，时间也越长
第6行:进行压缩的文件类型，建议js|css|图片,注:javascript有两种写法，最好都写上
第7行:跟squid等缓存服务有关，on的话会在Header里增加"Vary:Accept-Encoding"
第8行:IE6对Gzip不怎么友好，不给它Gzip了

gzip_vary off;是 Nginx 配置中的一条指令，它用于控制是否在响应头中设置 Vary:Accept-Encoding 头部，具体来说:
gzip_vary on;:启用时，Nginx 会在响应头中加入 vary:Accept-Encoding，表示不同的客户端可能会根据是否支持 gzip 压缩来收到不同的响应内容，这有助于缓存服务器(如CDN)缓存压缩和未压缩的内容。
gzip_vary off;:禁用时，Nginx 不会在响应头中加入 vary:Accept-Encoding，这意味着缓存服务器不会区分压缩与未压缩的内容。这样做通常会减少缓存的复杂性，但可能会导致某些情况缓存不一致。
```

**响应状态码**

200：正常响应

401：授权失败，比如账号、密码输入错误

403：权限不足，比如目录下找不到首页文件，权限不足都可能会引起 403

404：文件/资源未找到

500：服务器端异常，比如服务器端报错、数据库无法连接

502：gateway 网关错误，比如 nginx 转发请求到后端服务器，后端服务器异常，无法响应

## 日志参数详解

我们之前对 nginx.conf 进行了备份，可以打开备份看看日志的格式：

```shell
# vim /usr/local/nginx/conf/nginx.conf.bak
...
http {
    ...
    #log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
    #                  '$status $body_bytes_sent "$http_referer" '
    #                  '"$http_user_agent" "$http_x_forwarded_for"';
    #access_log  logs/access.log  main;
    ...
}

log_format main：表示设置日志的格式，及日志格式的名称是main
access_log logs/access.log  main; 表示设置访问日志的存放路径及使用名字是main的这个日志格式
```

| 参数 | 含义 |
| --- | --- |
| **$remote\_addr** | **客户端的 IP 地址** |
| $remote\_user | 用户记录远程客户端的用户名称（一般为"-"） |
| $time\_local | 用户记录访问时间和时区 |
| **$request** | **用户记录请求的 url 及请求方式** |
| $status | 响应状态码，比如：200 成功、404 页面找不到等 |
| $body\_bytes\_sent | 给客户端发送的文件主体内容字节数 |
| **$http\_user\_agent** | **用户所使用的代理（一般为服务器的厂商，比如：谷歌、火狐等）** |
| **$http\_x\_forwarded\_for** | **可以记录客户端 IP，通过代理服务器来记录客户端的 IP 地址，有负载均衡时记录** |
| $http\_referer | 可以记录用户是从哪个链接访问过来的 |

如果没有使用负载均衡 `$remote_addr` 就是客户端真实的 IP 地址，如果使用了负载均衡 `$http_x_forwarded_for` 就是客户端真实的 IP 地址。


在配置 nginx.conf 的时候，有一项是指定错误日志的，默认情况下你不指定也没有关系，因为 nginx 很少有错误日志记录的。但有时出现问题时，是有必要记录一下错误日志的，方便我们排查问题。

error\_log 级别分为 debug，info，notice，warn，error，crit 默认为 notice，该级别在日志名后边定义，格式如下：

| 日志级别（由小到大） | 含义 |
| --- | --- |
| debug | 调试模式，记录的日志最多 |
| info | 信息 |
| **notice** | **提示，默认 notice，大于等于 notice 级别的日志都会被记录** |
| warn | 警告 |
| error | 错误 |
| crit | 严重、致命，记录的日志最少 |

crit 记录的日志最少，而 debug 记录的日志最多。如果你的 nginx 遇到一些问题，比如 502 比较频繁出现，但是看默认的 error\_log 并没有看到有意义的信息，那么就可以调一下错误日志的级别，当你调成error 级别时，错误日志记录的内容会更加丰富。

小结:

错误日志的作用：用来查看错误信息，通过提示的错误信息，排除错误。

## 日志轮转（重点）

作用：利用 shell 脚本对日志进行切割，把大的访问日志切割为一个一个小日志，方便后期存储以及查看分析

```shell
编写日志轮转的脚本
# vim logrotate.sh
#!/bin/bash
date_info=$(date +%F-%H-%M)
mv /usr/local/nginx/logs/access.log /usr/local/nginx/logs/access-$date_info.log
/usr/local/nginx/sbin/nginx -s reload 或者用 systemctl reload nginx（取决于目前你用的哪种方式启动的Nginx！）

编写计划任务
# crontab -e
* */6 * * * /bin/sh /root/logrotate.sh &> /dev/null

执行脚本，可以先试一下
# bash /root/logrotate.sh
```

说明：为什么备份日志文件以后，要重载 Nginx 呢？`/usr/local/nginx/sbin/nginx -s reload`

答：因为每次重载，系统会判断是否有 access.log 文件，如果没有就会生成一个，如果有就不会生成了。

上面我们日志分割是将 access.log 重命名了，所以就不存在这个日志文件了，需要重载一下 Nginx 才会生成新的日志文件！

## location 的作用

location 指令的作用是根据用户请求的 uri 来执行不同的操作，也就是根据用户请求的网站 url 进行匹配，匹配指定的请求 uri（请求 uri 不包含查询字符串，如：http://localhost:8080/test?id=10，请求 uri 是 /test）

http:// => http 协议，https 协议（https 相比于 http 多了一个 s，安全证书，有了证书配置以后，传输就采用加密传输）

url：整体称之为 url => www.devops.com/test?id=10

uri：域名后面的路径，一般就称为 uri => /test

?id=10：代表参数

## location 优先级

优先级：location 如果有多个规则，则其也有优先级概念

```shell
优先级顺序：
1. 优先级最高
location = / {
    # 配置段
}

2. 优先匹配
loction ^~ /images/ {
    # 配置段
}

3. 区分大小写以及不区分大小写匹配网站资源
location ~* \.(gif|jpg|jpeg|png) {
    # 配置段
}

3. 根据资源目录进行匹配
location /documents/ {
    # 配置段
}

4. 当所有匹配对着都不满足时，匹配默认的location /
location / {
    # 配置段
}
```

> 注意：在一个 server 区块中，往往会有多个 location 匹配规则，只要匹配了其中一个，就不会继续匹配其他的规则了，如果所有的规则都不匹配，就会执行 location / 这个规则！

## server 优先级说明

```shell
server {
    listen 80;
    server_name localhost;
    root html;
    location / {
        index index.html index.htm;
    }
}
server {
    listen 80;
    server_name www.devops.com;
    root html/devops;
    location / {
        index index.html index.htm;
    }
}
```

观察上面的配置，有两个 server 块，server 块是根据 server\_name 来匹配请求的！！！

请求能匹配到某个 server 块后，就不会匹配其他的 server 块了！

如果请求匹配不到某个 server 块，那么就会默认使用第一个 server 块（也就是默认认为匹配到第一个 server 块！）

比如：

请求 url：http://www.devops.com，会匹配执行第二个 server 块

请求 url：http://192.168.126.174，第一个 server 块不匹配，第二个 server 块也不匹配，那么只能走第一个 server 块了！

> server 匹配到之后，然后再看该请求符合 server 中的哪个 location 规则！

# 六、URL 重写

## return

返回服务器的状态码，基本语法：

```shell
return 服务器响应状态码

200 OK，请求成功，正常响应
301 Moved Permanently，永久重定向，常用于重写规则
302 Found，临时重定向，常用于重写规则
304 Not Modified，请求资源内容没有改变，常用于缓存
404 Not Found，代表请求资源不存在，文件或路径不存在
500 Internal Server Error，服务器遇到了不知道如何处理的情况，常见于代码异常
502 Bad Gateway，错误网关，常出现在反向代理中，如果反向代理的服务器出现异常或者服务中断，会出现该报错
503 Service Unavailable，服务器没有准备好处理请求。常见于服务器因维护或重载而停机
```

请求状态码：

<font style="color:#117CEE;">https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Status</font>

永久重定向：一般用于域名变更，访问旧的域名时会重定向到新的域名

旧的域名：www.old.com

新的域名：www.new.com

简单测试：关键是记住上面的响应状态码表示什么含义！！！

```shell
server {
    listen 80;
    server_name localhost;
    root html;
    location = /301 {
        return 301;
    }
    location = /302 {
        return 302;
    }
    location / {
        index index.html index.htm;
    }
}
```

## rewrite

基本语法：

```shell
rewrite 匹配内容 替代内容 flag标记
```

flag 标记说明：

| 标记 | 作用 |
| --- | --- |
| last | 本条规则匹配完成之后，继续向下匹配新的 location URL 规则 |
| break | 本条规则匹配完成即终止，不再匹配后面的任何规则 |
| redirect | 返回 302 临时重定向，浏览器地址栏会显示跳转后的 url 地址 |
| permanent | 返回 301 永久重定向，浏览器地址栏会显示跳转后的 url 地址 |

> 注：多条 rewrite，从上到下匹配，匹配到之后就不再匹配其他 rewrite 规则。

另外还要特别注意：

last 与 break，进行重定向时，其客户端 URL 地址不会改变。

redirect 与 permanent，进行重定向时，其客户端 URL 地址会发生改变。

官方文档地址: [<font style="color:#117CEE;">http://nginx.org/en/docs/http/ngx</font>](http://nginx.org/en/docs/http/ngx)<font style="color:#117CEE;">\_http\_rewrite\_module.html#rewrite</font>

# 七、第三方模块

## fancyindex

nginx 本身官方没有的功能，一些开源开发者定制开发了，把代码公布出来，可以通过加载第三方模块的方式，使用新的功能。

第三方模块网址：https://docs.nginx.com/nginx/admin-guide/dynamic-modules/dynamic-modules/#nginx-plus-certified-modules

fancyindex 模块的作用就类似于之前我们学习的 autoindex on 的功能。表示列表展示出目录中的文件信息。只不过是该模块的功能更强大一些。

编译安装第三方模块：

```shell
上传两个第三方模块的安装包
# tar -xf ngx-fancyindex-0.5.2.tar.gz
# tar -xf echo-nginx-module-0.63.tar.gz

# cd /root/nginx-1.26.2/
# ./configure --prefix=/usr/local/nginx --user=www --group=www --with-http_ssl_module --with-http_stub_status_module --with-http_realip_module --add-module=/root/ngx-fancyindex-0.5.2/ --add-module=/root/echo-nginx-module-0.63

# make && make install && make upgrade
```

使用 fancyindex 模块，修改配置文件即可：

```shell
# cd /usr/local/nginx/
# vim conf/nginx.conf
server {
    listen 80;
    server_name localhost;
    root html;
    location / {
        fancyindex on;								# 使用fancyindex模块
        fancyindex_exact_size off;		# 
        index index;
    }
}
```

重启 Nginx，然后访问测试：

```shell
# systemctl restart nginx
```

## echo

作用：echo 模块常用来进行调试使用，比如：输出打印 Nginx 默认系统变量等。

```shell
# vim conf/nginx.conf
server {
    listen 80;
    server_name localhost;
    root html;
    location / {
        default_type text/plain; # 表示文本类型
        echo $document_root;
    }
}

# systemctl restart nginx
```

# 八、Nginx 安全管理

## 反向代理

实现隐藏真实服务的操作，起到一定安全作用（负载均衡章节讲解）

Nginx 可以隐藏后端真实服务器(保护数据)。

## 隐藏版本号

前置知识点：curl 命令

```shell
# curl http://192.168.126.174，发起http请求，得到响应内容
/usr/local/nginx/html

或者

# curl -I http://192.168.126.174，发起http请求，不获取响应内容，而获取响应信息
HTTP/1.1 200 OK
Server: nginx/1.26.2
Date: Sat, 29 Nov 2025 06:58:10 GMT
Content-Type: text/plain
Connection: keep-alive
```

由上图可知，Server: nginx/1.26.2 显示了服务器是 Nginx，并且版本为 1.26.2。每个版本都不是完美的，都可以存在漏洞，黑客可能会根据你的版本信息，攻击该版本出现的漏洞。

所以，我们最好将服务器的版本进行隐藏。

```shell
直接在http区块加入server_tokens off
http {
    server_tokens off;
    ...
}

# systemctl reload nginx

# curl -I http://192.168.126.174
HTTP/1.1 200 OK
Server: nginx			=>  可以看到已经不显示Nginx的版本号了！
Date: Sat, 29 Nov 2025 07:01:41 GMT
Content-Type: text/plain
Connection: keep-alive
```

## 配置 Nginx

`vim /usr/local/nginx/conf/nginx.conf`

> 之前我们改过 nginx 的配置文件中的许多内容！为了避免其他问题，大家可以将下面的内容直接覆盖到你自己的 nginx 配置文件中！

```shell
worker_processes  1;
events {
    worker_connections  1024;
}
http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile        on;
    keepalive_timeout  65;
    server {
        listen       80;
        server_name  localhost;
        root   html;
        location / {
            index  index.html index.htm;
        }

        # 配置Nginx的自动监控，Nginx本身自带了一些监控
        location = /stub_status {
            stub_status on;	# 开启Nginx的状态监控
            access_log off; # 监控产生的信息不记录到日志中
        }

        location ~ \.php$ {
          	fastcgi_pass   127.0.0.1:9000;
          	fastcgi_index  index.php;
          	fastcgi_param  SCRIPT_FILENAME  $document_root$fastcgi_script_name;
          	include        fastcgi_params;
        }
        
        error_page   500 502 503 504  /50x.html;
        location = /50x.html {
            root   html;
        }
    }
}
```

重启/重载 Nginx：
