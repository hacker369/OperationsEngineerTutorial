```shell
说明：
vm.overcommit_memory
这个参数是 Linux 系统在分配内存时的一种策略控制，直接关系到 Redis 这样的应用能不能顺利启动、正常运行。
默认值为0：系统会根据当前内存使用情况、总内存和交换空间的比例，"评估"是否允许分配内存。如果它“觉得”你分配太多内存了，就拒绝了，哪怕物理内存其实还够，Redis 启动时会尝试中请大量内存，比如它的最大内存限制值(maxmemory)，即使暂时还没用到。这时 Linux 系统”评估"你内存不够，就会阻止 Redis 启动，报错。

参数值设为1：系统不管三七二十一，你说要分配内存，它就答应。哪怕内存其实根本不够，它也会先允许你申请下来。如果设置为 1，系统允许进程分配“看起来够但实际上不够”的内存。如果很多程序都这么做，等真正用内存时可能会爆掉系统(内存溢出或00M)，但 Redis 一般是你可控的服务，设置maxmemory限制总量就没事.
Redis 推荐设置 vm.overcommit_memory = 1，就是为了避免系统“瞎担心内存不够”导致 Redis 启动失败。
```

## <font style="color:rgb(51, 51, 51);">key（键名）</font>

<font style="color:rgb(51, 51, 51);">内存：NoSQL 数据库，存储形式，键值对，类似身份证（不能重复，必须唯一）</font>

<font style="color:rgb(51, 51, 51);">key 的命名规则不同于一般语言，键盘上除了空格、\n 换行符外其他的大部分字符都可以使用。</font>

<font style="color:rgb(51, 51, 51);">但是像"my key"和"mykey\n"这样包含空格和换行的 key 是不允许的。</font>

<font style="color:rgb(51, 51, 51);">我们在使用的时候可以自己定义一个key的格式，但是要特别注意：</font>

<font style="color:rgb(51, 51, 51);">key 不要太长。占内存，查询慢。</font>

<font style="color:rgb(51, 51, 51);">key 不要太短。像 u:1000:pwd:123456 就不如 user:1000:password:123456 可读性好</font>

## <font style="color:rgb(51, 51, 51);">list</font>

<font style="color:rgb(51, 51, 51);">key value(value1,value2,value3)</font>

<font style="color:rgb(51, 51, 51);">list 类型其实就是一个双向链表。通过 push，pop 操作从链表的头部或者尾部添加删除元素</font>

<font style="color:rgb(51, 51, 51);">这使得 list 既可以用作栈，也可以用作队列</font>

<font style="color:rgb(51, 51, 51);">同一端进出，先进后出，后进先出 ==> 栈</font>

### <font style="color:rgb(51, 51, 51);">RDB优劣势</font>

**<font style="color:rgb(51, 51, 51);">优点：</font>**

<font style="color:rgb(51, 51, 51);">RDB 是 Redis 数据的一个非常紧凑的单文件时间点表示。RDB 文件非常适合备份。例如，您可能想要对最近 24 小时内的 RDB 文件进行每小时存档，并保存30天内每天的RDB快照。这允许您在发生灾难时轻松地恢复数据集的不同版本。</font>

<font style="color:rgb(51, 51, 51);">a) 适合大规模数据恢复</font>

<font style="color:rgb(51, 51, 51);">b) 按照业务定时备份</font>

<font style="color:rgb(51, 51, 51);">c) 对数据的完整性和一致性要求不高</font>

<font style="color:rgb(51, 51, 51);">d) RDB 文件在内存中加载速度要比 AOF 快很多。</font>

**<font style="color:rgb(51, 51, 51);">缺点：</font>**

<font style="color:rgb(51, 51, 51);">a) 需要在一定间隔时间做一次备份，所以如果 Redis 意外 down 掉的话，就会丢失从当前至最近一次快照期间，还未来得及保存的数据。丢失最新数据的准备。</font>

<font style="color:rgb(51, 51, 51);">b) 内存数据的全量同步，如果数据量太大会导致I/O严重影响服务器性能。例如：1天修改1000次，同步一下数据。同步一次，数据量会太大。</font>

<font style="color:rgb(51, 51, 51);">c) 为了使用子进程持久化到磁盘上，RDB 经常需要 fork()。如果数据集很大，fork() 可能很耗时，如果数据集很大，CPU 性能不太好，可能会导致 Redis 停止为客户端服务几毫秒甚至一秒钟。AOF 也需要fork()，但不太频繁，你可以调整重写日志的频率，而无需权衡持久性。</font>

### <font style="color:rgb(51, 51, 51);">修复dump.rdb文件</font>

说明：正常情况下，使用 RDB 方式进行数据备份后，会放在我们之前在配置文件中指定的位置，名字为 xxx.dump。之后停掉 redis，当我们再次启动 redis 的时候，redis 就会根据配置文件中配置的 xxx.dump 的路径去找 xxx.dump 文件进行数据的恢复！

但是，有时候，我们将 redis 的备份文件 dump.rdb 从指定的位置移动到另一个位置，然后再移动回来，可能通过该备份文件就恢复不了数据了，这个备份文件就被损坏了！

**怎样修复 dump.rdb 文件？**

```shell
# redis-check-rdb [dump.rdb文件名称]
```

修复完 dump.rdb 文件后，将文件移动到配置文件指定的备份路径，然后启动 redis，看是否恢复成功！

### <font style="color:rgb(51, 51, 51);">小结</font>

![1765270413430-38e4bd7d-34a6-4af6-868c-48c781b6cd87.png](./img/I5IN2Jv85tX6eYZY/1765270413430-38e4bd7d-34a6-4af6-868c-48c781b6cd87-280974.png)

**<font style="color:rgb(51, 51, 51);">优势：</font>**

<font style="color:rgb(51, 51, 51);">RDB是一个非常紧凑的文件</font>

<font style="color:rgb(51, 51, 51);">RDB在保存RDB文件时父进程唯一需要做的就是fork出一个子进程，接下来的工作全部由子进程来做，父进程不需要再做其他IO操作，所以RDB持久化方式可以最大化redis的性能。与AOF相比，在恢复大的数据集的时候，RDB方式会更快一些。</font>

**<font style="color:rgb(51, 51, 51);">缺点：</font>**

<font style="color:rgb(51, 51, 51);">数据丢失风险大</font>

<font style="color:rgb(51, 51, 51);">RDB 需要经常fork子进程来保存数据集到硬盘上，当数据集比较大的时候，fork的过程是非常耗时的，可能会导致Redis在一些毫秒级不能响应客户端请求。</font>

## <font style="color:rgb(51, 51, 51);">append only file(aof)</font>

<font style="color:rgb(51, 51, 51);">相同点：RDB 和 AOF 都是 Redis 持久化的一种方式。</font>

<font style="color:rgb(51, 51, 51);">不同点：RDB 采用快照持久化(全量备份)、AOF(Append only file)把用户对 Redis 的 DML 操作(事务型操作)追加到 AOF 文件中</font>

### <font style="color:rgb(51, 51, 51);">AOF介绍</font>

**<font style="color:rgb(51, 51, 51);">AOF (Append Only File) :</font>**<code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 持久化记录服务器接收到的每个写操作。这些操作可以在服务器启动时重播，重建原始数据集。命令使用与Redis协议本身相同的格式记录。</font>

**<font style="color:rgb(51, 51, 51);">通俗易懂：</font>**<font style="color:rgb(51, 51, 51);">以日志的形式记录着每个</font>**<font style="color:rgb(51, 51, 51);">写</font>**<font style="color:rgb(51, 51, 51);">操作，将 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 执行过的所有</font>**<font style="color:rgb(51, 51, 51);">写</font>**<font style="color:rgb(51, 51, 51);">指令全都记录下来（读操作不记录），只许追加文件但不可以改写文件。Redis启动之初会读取该文件重新构建数据。换言之，Redis重启的话就根据日志文件的内容将</font>**<font style="color:rgb(51, 51, 51);">写</font>**<font style="color:rgb(51, 51, 51);">指令从前到后执行一次以完成数据的恢复工作（类似relay log中继日志效果）。</font>

### <font style="color:rgb(51, 51, 51);">AOF生成文件说明</font>

<font style="color:rgb(51, 51, 51);">Redis 6或之前版本：AOF保存文件名称有且只有一个</font>

<font style="color:rgb(51, 51, 51);">Redis 7：采用了 Redis 7.0 Multi Part AOF 的设计，</font><code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font></code><font style="color:rgb(51, 51, 51);">保存文件名称从1到3。分别是：</font>**<font style="color:rgb(51, 51, 51);">base基础文件</font>**<font style="color:rgb(51, 51, 51);"> ， </font>**<font style="color:rgb(51, 51, 51);">incr增量文件</font>**<font style="color:rgb(51, 51, 51);"> ，</font>**<font style="color:rgb(51, 51, 51);">manifest清单文件</font>**<font style="color:rgb(51, 51, 51);"> 。</font>

![1765275698713-befd25de-6fa0-49e1-bb1d-4469fbb556e7.png](./img/I5IN2Jv85tX6eYZY/1765275698713-befd25de-6fa0-49e1-bb1d-4469fbb556e7-533585.png)

<font style="color:rgb(51, 51, 51);">MP-AOF实现：</font><font style="color:rgb(51, 51, 51);">顾名思义， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">MP-AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 就是将原来的单个 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 文件拆分成多个AOF文件。在 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">MP-AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 中，我们将 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 分为三种类型：</font>

* **<font style="color:rgb(51, 51, 51);">BASE：</font>**<font style="color:rgb(51, 51, 51);"> 表示基础的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> ，一般由子进程通过重写产生，该文件最多只有一个。</font>
* **<font style="color:rgb(51, 51, 51);">INCR：</font>**<font style="color:rgb(51, 51, 51);"> 表是增量的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> ，一般由 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOFRW</font>**</code><font style="color:rgb(51, 51, 51);"> 开始执行的时候被创建，该文件可能存在多个。</font>
* **<font style="color:rgb(51, 51, 51);">HISTORY：</font>**<font style="color:rgb(51, 51, 51);"> 表是历史的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> ，由 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">BASE</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">INCR</font>**</code><font style="color:rgb(51, 51, 51);"> 的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 变化而来，每次 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOFRW</font>**</code><font style="color:rgb(51, 51, 51);"> 成功完成时，本次 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOFRW</font>**</code><font style="color:rgb(51, 51, 51);"> 之前对应的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">BASE</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">INCR</font>**</code><font style="color:rgb(51, 51, 51);"> 的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 都将变成 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">HISTORY</font>**</code><font style="color:rgb(51, 51, 51);"> ， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, ### <font style="color:rgb(51, 51, 51);">AOF重写机制</font>

<font style="color:rgb(51, 51, 51);">AOF 一共生成3个文件，base 基础文件，incr 增量文件，manifest 清单文件（记录本次备份都有哪些文件）</font>

<font style="color:rgb(51, 51, 51);">AOF 实际经常操作的只有一个 incr 增量文件，随着时间增长越来越大，为了解决这个问题，我们可以使用AOF 重写来减小文件大小。</font>

***

<font style="color:rgb(51, 51, 51);">由于 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 持久化是 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 不断将写命令记录到 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 中，随着 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 不断的进行，</font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 的文件会越来越大，文件越大，占用服务器内存越大以及 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 恢复要求时间越长。</font><font style="color:rgb(51, 51, 51);">为了解决这个问题， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code>**<font style="color:rgb(51, 51, 51);">新增了重写机制</font>**<font style="color:rgb(51, 51, 51);">，当 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 的大小超过所设定的峰值时， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 就会 </font>**<font style="color:rgb(51, 51, 51);">自动</font>**<font style="color:rgb(51, 51, 51);"> 启动 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 的内容压缩，只保留可以恢复数据的最小指令集或者可以手动使用命令 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">bgrewriteaof</font>**</code><font style="color:rgb(51, 51, 51);">。</font>

<font style="color:rgb(51, 51, 51);">触发机制：打开 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 的配置文件 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">redis.conf</font>**</code>

![1765276914584-ea462b9f-6979-4fa5-b1b9-d58f9e59e4cc.png](./img/I5IN2Jv85tX6eYZY/1765276914584-ea462b9f-6979-4fa5-b1b9-d58f9e59e4cc-654496.png)

<font style="color:rgb(51, 51, 51);">参数解析：</font>

<font style="color:rgb(51, 51, 51);">① 假设 auto-aof-rewrite-min-size 设置为 64mb，auto-aof-rewrite-percentage 设置为 100。当 AOF文件首次达到 64mb 时，Redis 会进行第一次重写。</font>

<font style="color:rgb(51, 51, 51);">② 假设新生成的 AOF 文件大小为 64mb。之后，当 AOF 文件再次增长到 128mb（即 64mb 的 200%）时，由于当前 AOF 大小超过了上次重写后的 100%（64mb \* 1 +64mb=128mb），并且也超过了 min-size 的 64mb，此时就会触发第二次重写。</font>

**<font style="color:rgb(51, 51, 51);">自动触发：</font>**<font style="color:rgb(51, 51, 51);">除了第一次外</font>**<font style="color:rgb(51, 51, 51);">，</font>**<font style="color:rgb(51, 51, 51);">之后要触发必须同时满足上面两个条件，才会触发重写机制。官网默认 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 是 </font>**<font style="color:rgb(51, 51, 51);">64M</font>**<font style="color:rgb(51, 51, 51);"> ，可以手动修改大小。</font>

**<font style="color:rgb(51, 51, 51);">手动触发：</font>**<font style="color:rgb(51, 51, 51);">客户端向服务端发送如下命令 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">bgrewriteaof</font>**</code><font style="color:rgb(51, 51, 51);"> 即可手动触发。</font>

***

<font style="color:rgb(51, 51, 51);">AOF 文件重写并不是对原文件进行重新整理，而是直接读取服务器现有的键值对，然后用一条命令去代替之前记录的这个键值对的多条命令，生成一个新的文件后替换原来的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 。</font>

**<font style="color:rgb(51, 51, 51);">AOF 重写触发机制：</font>**<font style="color:rgb(51, 51, 51);"> 通过 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">redis.conf</font>**</code><font style="color:rgb(51, 51, 51);"> 配置文件中的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">auto-aof-rewrite-percentage：默认值为100</font>**</code><font style="color:rgb(51, 51, 51);"> ，以及 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">auto-aof-rewrite-min-size：默认值为64mb</font>**</code><font style="color:rgb(51, 51, 51);"> 。也就是说，默认 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 会记录上一次重写时的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 大小，默认配置是当 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 大小是上次 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">rewrite（重写）</font>**</code><font style="color:rgb(51, 51, 51);"> 后大小的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">1</font>**</code><font style="color:rgb(51, 51, 51);"> 倍且文件大于 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">64MB</font>**</code><font style="color:rgb(51, 51, 51);"> 时触发。</font>

#### <font style="color:rgb(51, 51, 51);">重写原理</font>

<font style="color:rgb(51, 51, 51);">① 在重写开始前， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 会创建一个 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">重写子进程</font>**</code><font style="color:rgb(51, 51, 51);"> ，这个子进程 </font>**<font style="color:rgb(51, 51, 51);">不会</font>**<font style="color:rgb(51, 51, 51);"> 读取现有的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> ，而是读取当前服务器上数据库内容，并将其包含的指令进行分析压缩并写入到一个临时文件中。</font>

<font style="color:rgb(51, 51, 51);">② 与此同时，主进程会将新接收到的 </font>**<font style="color:rgb(51, 51, 51);">写指令</font>**<font style="color:rgb(51, 51, 51);"> 一边累积到内存缓冲区中，一边继续写入到原有的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 中，这样做是保证原有的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 的可用性，避免在重写过程中出现意外。</font>

<font style="color:rgb(51, 51, 51);">③ 当 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">重写子进程</font>**</code><font style="color:rgb(51, 51, 51);"> 完成重写工作后，它会给父进程发一个信号，父进程收到信号后就会将内存中缓存的写指令 </font>**<font style="color:rgb(51, 51, 51);">追加</font>**<font style="color:rgb(51, 51, 51);"> 到新 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 中。</font>

<font style="color:rgb(51, 51, 51);">④当追加结束后， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 就会用 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">新AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 来 </font>**<font style="color:rgb(51, 51, 51);">替换</font>**<code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">旧AOF文件</font>**</code><font style="color:rgb(51, 51, 51);">，之后再有新的写指令，就都会追加到 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">新AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 中。</font>

<font style="color:rgb(51, 51, 51);">⑤ 重写 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> 的操作，并没有读取 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">旧AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> ，而是将整个内存中的数据库内容用命令的方式重写了一个 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">新AOF文件</font>**</code><font style="color:rgb(51, 51, 51);"> ，这点和快照有点类似。</font>

### <font style="color:rgb(51, 51, 51);">小结</font>

![1765277789405-87e17838-9d2e-4e8c-a27c-5a04784e9948.png](./img/I5IN2Jv85tX6eYZY/1765277789405-87e17838-9d2e-4e8c-a27c-5a04784e9948-446912.png)

**<font style="color:rgb(51, 51, 51);">优势：</font>**

<font style="color:rgb(51, 51, 51);">AOF 文件是一个只进行追加的日志文件</font>

<font style="color:rgb(51, 51, 51);">Redis 可以在 AOF 文件体积变得过大时，自动地在后台对 AOF 进行重写</font>

<font style="color:rgb(51, 51, 51);">AOF 文件有序地保存了对数据库执行的所有写入操作，这些写入操作以 Redis 协议的格式保存，因此 AOF文件的内容非常容易被人读懂，对文件进行分析也很轻松</font>

**<font style="color:rgb(51, 51, 51);">缺点：</font>**

<font style="color:rgb(51, 51, 51);">对于相同的数据集来说，AOF 文文件的体积通常要大于 RDB 文件的体积。</font>

<font style="color:rgb(51, 51, 51);">根据所使用的 fsync 策略，AOF 的速度可能会慢于 RDB</font>

## <font style="color:rgb(51, 51, 51);">扩展：No persistence（了解）</font>

**<font style="color:rgb(51, 51, 51);">No persistence :</font>**<font style="color:rgb(51, 51, 51);"> 完全禁用持久性。这有时用于缓存，属于纯净模式。</font>

<font style="color:rgb(51, 51, 51);">需要同时禁用 RDB 和 AOF</font>

**<font style="color:rgb(51, 51, 51);">注：</font>**<code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 禁用，只是单纯的禁用自动触发。不会禁用手动触发，手动输入命令，仍然可以触发 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code>

### <font style="color:rgb(51, 51, 51);">禁用RDB</font>

<font style="color:rgb(51, 51, 51);">配置文件： </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">save ""</font>**</code><font style="color:rgb(51, 51, 51);">禁用 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 模式下，仍然可以手动输入命令 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">save</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">bgsave</font>**</code><font style="color:rgb(51, 51, 51);"> 来生成 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 文件。</font>

![1765281086586-aeec7ada-980a-477c-b879-755b6fee1916.png](./img/I5IN2Jv85tX6eYZY/1765281086586-aeec7ada-980a-477c-b879-755b6fee1916-900785.png)

### **<font style="color:rgb(51, 51, 51);">禁用AOF</font>**

<font style="color:rgb(51, 51, 51);">配置文件： </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">appendonly no</font>**</code>

<font style="color:rgb(51, 51, 51);">禁用 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 模式下，仍然可以手动输入命令 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">bgrewriteaof</font>**</code><font style="color:rgb(51, 51, 51);"> 来生成 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 文件。</font>

![1765281118869-be5b16fc-94cc-4ef2-a133-272de8ab3b36.png](./img/I5IN2Jv85tX6eYZY/1765281118869-be5b16fc-94cc-4ef2-a133-272de8ab3b36-024069.png)

## <font style="color:rgb(51, 51, 51);">扩展：RDB + AOF（生产环境）</font>

**<font style="color:rgb(51, 51, 51);">RDB + AOF :</font>**<font style="color:rgb(51, 51, 51);"> 在同一个实例中同时使用 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code>

### <font style="color:rgb(51, 51, 51);">共存优先级</font>

![1765281321929-25811050-6df0-4e99-93e0-582ae22cfd3a.png](./img/I5IN2Jv85tX6eYZY/1765281321929-25811050-6df0-4e99-93e0-582ae22cfd3a-557550.png)

<code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 持久化可以同时启用，不会有问题。如果启动时启用了 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> ， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 将加载 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> ，即文件具有更好的持久性保证。则 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 的优先级高于 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code>

### <font style="color:rgb(51, 51, 51);">数据恢复顺序及加载流程</font>

<font style="color:rgb(51, 51, 51);">同时开启 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 时， </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">Redis</font>**</code><font style="color:rgb(51, 51, 51);"> 默认优先加载 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> ，则不去加载 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> （可以说和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 毫无关系）。如果 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 文件不存在，再去加载 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code>

![1765281360851-3c1fd5d5-4e69-4446-8e9c-cf38ee5c34bd.png](./img/I5IN2Jv85tX6eYZY/1765281360851-3c1fd5d5-4e69-4446-8e9c-cf38ee5c34bd-831995.png)

### <font style="color:rgb(51, 51, 51);">开启RDB和AOF混合模式</font>

<font style="color:rgb(51, 51, 51);">结合了 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 的优点，既能快速加载又能避免丢失过多的数据。江湖人称“ </font>**<font style="color:rgb(51, 51, 51);">鸳鸯锅</font>**<font style="color:rgb(51, 51, 51);"> ”</font>

<font style="color:rgb(51, 51, 51);">配置文件中 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">aof-use-rdb-preamble</font>**</code><font style="color:rgb(51, 51, 51);"> 的默认值为 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">yes</font>**</code><font style="color:rgb(51, 51, 51);"> 。 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">yes</font>**</code><font style="color:rgb(51, 51, 51);"> 表示开启，设置为 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">no</font>**</code><font style="color:rgb(51, 51, 51);"> 表示禁用。</font>

![1765281393849-9e19d709-d4b0-464c-82d1-750b8a606b89.png](./img/I5IN2Jv85tX6eYZY/1765281393849-9e19d709-d4b0-464c-82d1-750b8a606b89-714214.png)

<code><font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB+AOF</font></code><font style="color:rgb(51, 51, 51);"> 的混合方式</font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 镜像做全量持久化，</font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 做增量持久化。先使用 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 进行快照存储，然后使用 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 持久化记录所有的写操作。当重写策略满足或手动触发重写的时候，将最新的数据存储为新的 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 记录。这样的话，重启服务的时候会从 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 和 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 两部分恢复数据，既保证了数据完整性，又提高了恢复数据的性能。</font>

<font style="color:rgb(51, 51, 51);">简单来说：混合持久化方式产生的文件一部分是 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 格式，一部分是 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 格式。即 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 包括了 </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">RDB</font>**</code><font style="color:rgb(51, 51, 51);"> 头部 + </font><code>**<font style="color:rgb(51, 51, 51);background-color:rgb(243, 244, 244);">AOF</font>**</code><font style="color:rgb(51, 51, 51);"> 混写。</font>

![1765281433449-b2712a36-1477-4387-b967-0e03297d5ab8.png](./img/I5IN2Jv85tX6eYZY/1765281433449-b2712a36-1477-4387-b967-0e03297d5ab8-163772.png)

## <font style="color:rgb(51, 51, 51);">总结</font>

<font style="color:rgb(51, 51, 51);">掌握：学会开启与恢复RDB、学会开启与恢复AOF即可！！！</font>

***

<font style="color:rgb(51, 51, 51);">rdb 适用于一般的数据持久化使用，效率高，数据迁移方便</font>

<font style="color:rgb(51, 51, 51);">aof 适合于增量备份、数据实时性备份要求更高的情况</font>

<font style="color:rgb(51, 51, 51);">rdb和aof同时开启，redis听谁的？</font>

<font style="color:rgb(51, 51, 51);">答：</font>

<font style="color:rgb(51, 51, 51);">① 听aof的，rdb与aof同时开启默认加载AOF的配置文件</font>

<font style="color:rgb(51, 51, 51);">② 相同数据集，aof文件要远大于rdb文件，恢复速度慢于rdb</font>

<font style="color:rgb(51, 51, 51);">③ aof运行效率慢于rdb，但是同步策略效率好，异步效率和rdb相同</font>

<font style="color:rgb(51, 51, 51);">概括：生产环境都是混合模式居多，RDB负责全量备份，AOF适合增量备份。RDB恢复数据快，文件小。</font>

<font style="color:rgb(51, 51, 51);">如果只能选一个：一般场景用RDB，实时要求高场景用AOF。</font>

```python
# mkdir -p /usr/local/lua

local function close_redis(red)
    if not red then
        return
    end
    --释放连接(连接池实现)
    local pool_max_idle_time = 10000 --毫秒
    local pool_size = 100 --连接池大小
    local ok, err = red:set_keepalive(pool_max_idle_time, pool_size)
 
    if not ok then
        ngx_log(ngx_ERR, "set redis keepalive error : ", err)
    end
end
 
local redis = require "resty.redis"
local red = redis:new()
red:set_timeout(1000)
local ip = "192.168.126.201" -- redis ip
local port = 6379 -- redis port
local ok, err = red:connect(ip,port)
red:auth("123") -- redis auth password
if not ok then
    return close_redis(red)
end
 
local clientIP = ngx.req.get_headers()["X-Real-IP"]
if clientIP == nil then
   clientIP = ngx.req.get_headers()["x_forwarded_for"]
end
if clientIP == nil then
   clientIP = ngx.var.remote_addr
end
 
local incrKey = "user:"..clientIP..":freq"
local blockKey = "user:"..clientIP..":block"
 
local is_block,err = red:get(blockKey) -- check if ip is blocked
if tonumber(is_block) == 1 then
   ngx.exit(ngx.HTTP_FORBIDDEN) -- return 403
   return close_redis(red)
end
 
res, err = red:incr(incrKey)
 
if res == 1 then
   res, err = red:expire(incrKey,1)
end
 
if res > 100 then	-- block 频率，测试环境可以考虑设置为3或5
    res, err = red:set(blockKey,1)
    res, err = red:expire(blockKey,600)    -- 设置ip屏蔽的时间，600秒
end
 
close_redis(red)
```
